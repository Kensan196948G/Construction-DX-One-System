## 1. リリース管理概要

### 1.1 目的

本プラットフォームのリリースプロセスを標準化し、品質を保証しながら安全かつ効率的にリリースを実施する。ISO20000変更管理プロセスに準拠したリリース管理を行う。

### 1.2 リリース方針

| 方針 | 内容 |
|------|------|
| リリース頻度 | 定期リリース（月1回）+ 緊急リリース（随時） |
| リリースウィンドウ | 毎月第3水曜日 22:00-02:00（翌日） |
| フリーズ期間 | 期末（3月/9月の最終2週間）はリリース禁止 |
| ロールバック方針 | 全リリースにロールバック手順を必須化 |
| 承認フロー | 本プラットフォーム自身のCABプロセスを利用（自己準拠） |

## 2. リリース戦略

### 2.1 リリース種別

| 種別 | 定義 | バージョン変更 | 承認 | リードタイム |
|------|------|-------------|------|------------|
| メジャーリリース | 大幅な機能追加、破壊的変更 | X.0.0 | CAB承認 + 経営承認 | 10営業日前 |
| マイナーリリース | 新機能追加、後方互換あり | x.Y.0 | CAB承認 | 5営業日前 |
| パッチリリース | バグ修正、セキュリティパッチ | x.y.Z | CAB承認（簡略化） | 3営業日前 |
| 緊急リリース | 重大バグ/脆弱性対応 | x.y.Z | ECAB承認 | 即時 |

### 2.2 リリーストレイン

```
Month 1       Month 2       Month 3       Month 4
  │             │             │             │
  ▼             ▼             ▼             ▼
 v1.1.0       v1.2.0       v1.3.0       v1.4.0
  │             │             │             │
  ├── v1.1.1    ├── v1.2.1    ├── v1.3.1    │
  └── v1.1.2    └── v1.2.2    │             │
                              │             │
                          （フリーズ期間）     │
                           リリース禁止      ▼
                                          v2.0.0（メジャー）
```

## 3. バージョニング規約

### 3.1 セマンティックバージョニング

```
MAJOR.MINOR.PATCH
  │     │     │
  │     │     └── バグ修正、後方互換あり
  │     └──────── 新機能追加、後方互換あり
  └────────────── 破壊的変更、後方互換なし
```

### 3.2 バージョン付与ルール

| 変更内容 | バージョン変更 | 例 |
|---------|-------------|-----|
| APIエンドポイント削除/変更 | MAJOR | 1.0.0 → 2.0.0 |
| DBスキーマの破壊的変更 | MAJOR | 1.0.0 → 2.0.0 |
| 新機能追加（後方互換あり） | MINOR | 1.1.0 → 1.2.0 |
| 既存機能の改善 | MINOR | 1.1.0 → 1.2.0 |
| バグ修正 | PATCH | 1.1.0 → 1.1.1 |
| セキュリティパッチ | PATCH | 1.1.0 → 1.1.1 |
| ドキュメント修正のみ | 変更なし | - |

### 3.3 プレリリースバージョン

| 段階 | 形式 | 用途 |
|------|------|------|
| アルファ | x.y.z-alpha.N | 内部テスト版 |
| ベータ | x.y.z-beta.N | ステージングテスト版 |
| リリース候補 | x.y.z-rc.N | リリース最終確認版 |

## 4. ブランチ戦略

### 4.1 ブランチモデル（GitHub Flow ベース）

```
main ─────────────────────────────────────────────────
  │           ▲           ▲               ▲
  │           │ merge     │ merge         │ merge
  │           │           │               │
  └── feature/rfc-form ───┘               │
  │                                       │
  └── feature/cab-approval ───────────────┘
  │
  └── hotfix/fix-auth-bug ──────────────── (緊急マージ)
```

### 4.2 ブランチ命名規則

| ブランチ種別 | 命名規則 | 例 |
|-------------|---------|-----|
| メインブランチ | `main` | main |
| 機能ブランチ | `feature/<機能名>` | feature/rfc-form-validation |
| バグ修正ブランチ | `fix/<バグ名>` | fix/cab-approval-error |
| 緊急修正ブランチ | `hotfix/<修正名>` | hotfix/security-patch |
| リリースブランチ | `release/v<バージョン>` | release/v1.2.0 |

### 4.3 ブランチ保護ルール

| ルール | main | release/* |
|-------|------|----------|
| プルリクエスト必須 | 必須 | 必須 |
| レビュー承認数 | 1名以上 | 2名以上 |
| ステータスチェック必須 | CI全パス | CI全パス + E2Eテスト |
| 強制プッシュ禁止 | 禁止 | 禁止 |
| ブランチ削除保護 | 保護 | 保護 |

## 5. リリースプロセス

### 5.1 通常リリースフロー

```
1. リリース計画策定（リリース10営業日前）
   │
2. フィーチャーフリーズ（リリース5営業日前）
   │  → 新機能のマージ停止、バグ修正のみ許可
   │
3. リリースブランチ作成（リリース3営業日前）
   │  → release/vX.Y.Z ブランチ作成
   │
4. ステージング環境デプロイ（リリース3営業日前）
   │  → E2Eテスト・性能テスト実施
   │
5. RFC起票・CAB承認（リリース3営業日前）
   │  → 本プラットフォームのCABで自己承認
   │
6. リリース承認（リリース1営業日前）
   │  → テスト結果確認、チェックリスト完了
   │
7. 本番デプロイ実施（リリース当日）
   │  → docker compose 更新
   │
8. スモークテスト（デプロイ後30分以内）
   │  → ヘルスチェック、主要機能動作確認
   │
9. リリース完了通知
   │
10. 事後レビュー（翌営業日）
```

### 5.2 緊急リリースフロー

```
1. 緊急修正ブランチ作成
   │  → hotfix/xxx をmainから作成
   │
2. 修正実装 + テスト
   │  → 単体テスト + 該当箇所の結合テスト
   │
3. ECAB緊急承認
   │  → ECAB承認フローで即時承認
   │
4. 本番デプロイ
   │
5. スモークテスト
   │
6. 事後RFC起票（翌営業日）
```

### 5.3 リリースチェックリスト

| カテゴリ | チェック項目 | 確認者 |
|---------|------------|--------|
| テスト | 全単体テスト合格 | 開発者 |
| テスト | 全結合テスト合格 | 開発者 |
| テスト | E2Eテスト合格（ステージング環境） | QA |
| テスト | 性能テスト目標値達成 | QA |
| テスト | セキュリティスキャン合格 | セキュリティ担当 |
| コード | コードレビュー完了 | レビュアー |
| コード | マイグレーションスクリプト確認 | DB管理者 |
| 運用 | ロールバック手順確認 | 運用チーム |
| 運用 | バックアップ取得完了 | 運用チーム |
| 運用 | メンテナンス通知送信済み | 運用チーム |
| 承認 | CAB承認取得 | CAB議長 |
| 承認 | リリース責任者承認 | IT部門長 |

## 6. リリースノート

### 6.1 リリースノート形式

```markdown

**リリース日:** 2026-04-15

## 新機能
- RFC起票フォームに影響分析自動実行機能を追加 (#123)
- CABダッシュボードにリアルタイム通知を追加 (#145)

## 改善
- カレンダー表示の応答速度を改善（平均1.2秒→0.8秒）(#150)
- RFC一覧のフィルタリングUIを改善 (#148)

## バグ修正
- フリーズ期間の終了日計算の不具合を修正 (#152)
- 通知メールの文字化け問題を修正 (#155)

## セキュリティ
- 依存パッケージの脆弱性対応（npm audit fix）

## 破壊的変更
- なし

## 既知の問題
- Internet Explorer 11 は非対応

## アップグレード手順
1. docker compose pull
2. docker compose up -d
3. マイグレーション実行: npm run migrate
```

## 7. タグ管理

### 7.1 Gitタグ規約

| タグ形式 | 用途 | 例 |
|---------|------|-----|
| `vX.Y.Z` | 正式リリース | v1.2.0 |
| `vX.Y.Z-rc.N` | リリース候補 | v1.2.0-rc.1 |
| `vX.Y.Z-beta.N` | ベータ版 | v1.2.0-beta.1 |

### 7.2 タグ作成手順

```bash
# リリースタグ作成
git tag -a v1.2.0 -m "Release v1.2.0: RFC影響分析機能追加"
git push origin v1.2.0

# GitHub Release 作成
gh release create v1.2.0 \
  --title "v1.2.0" \
  --notes-file RELEASE_NOTES.md
```
