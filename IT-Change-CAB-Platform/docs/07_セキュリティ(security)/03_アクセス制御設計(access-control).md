## 1. アクセス制御設計方針

### 1.1 基本原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | 最小権限の原則（Least Privilege） | ユーザには業務遂行に必要な最小限の権限のみを付与する |
| 2 | 職務分離（Separation of Duties） | 変更申請者と承認者を分離し、自己承認を禁止する |
| 3 | Need-to-Know | 業務上知る必要のある情報にのみアクセスを許可する |
| 4 | 暗黙的拒否（Deny by Default） | 明示的に許可されていないアクセスはすべて拒否する |
| 5 | 一時的権限付与 | 常設の広範な権限ではなく、必要時に一時的に付与する |

### 1.2 アクセス制御モデル

本プラットフォームでは、以下のアクセス制御モデルを組み合わせて使用する。

| モデル | 適用範囲 | 説明 |
|--------|---------|------|
| RBAC（ロールベースアクセス制御） | 主要なアクセス制御 | ロールに基づくAPIエンドポイント・機能への制御 |
| ABAC（属性ベースアクセス制御） | 補完的制御 | リソースの属性（所有者、部門、ステータス）に基づく制御 |
| オブジェクトレベルアクセス制御 | データアクセス | 個別リソース（RFC、CAB決定）単位でのアクセス制御 |

## 2. ロール定義

### 2.1 ロール一覧

| ロールID | ロール名 | 日本語名 | 説明 | 対象者 |
|----------|---------|---------|------|--------|
| CAB_CHAIR | CAB Chairman | CAB議長 | CABの議事進行と最終承認権限を持つ | IT部門長 |
| CAB_TECH | Technical Member | 技術メンバー | 技術的観点から変更を審議・承認する | IT部門員（3名） |
| CAB_BIZ | Business Member | ビジネスメンバー | 業務影響の観点から変更を審議する | 各部門代表 |
| SEC_REVIEWER | Security Reviewer | セキュリティレビュア | セキュリティ観点での変更レビューを担当 | IT部門セキュリティ担当 |
| EXEC_APPROVER | Executive Approver | 経営承認者 | 重大変更の最終経営判断・承認 | 経営層 |
| CHANGE_REQ | Change Requester | 変更申請者 | RFC（変更申請）の起票と管理 | IT部門員、各部門担当者 |
| CHANGE_IMPL | Change Implementer | 変更実施者 | 承認済み変更の実施と記録 | IT部門員 |
| VIEWER | Viewer | 閲覧者 | 読み取り専用アクセス | 全従業員 |
| SYS_ADMIN | System Administrator | システム管理者 | プラットフォーム全体の設定・管理 | IT部門管理者（2名） |

### 2.2 ロール階層

```
SYS_ADMIN（システム管理者）
  │
  ├── CAB_CHAIR（CAB議長）
  │     ├── CAB_TECH（技術メンバー）
  │     ├── CAB_BIZ（ビジネスメンバー）
  │     └── SEC_REVIEWER（セキュリティレビュア）
  │
  ├── EXEC_APPROVER（経営承認者）
  │
  ├── CHANGE_REQ（変更申請者）
  │     └── CHANGE_IMPL（変更実施者）
  │
  └── VIEWER（閲覧者）← 全ロールはVIEWER権限を継承
```

### 2.3 ロールの複数割当て

一人のユーザに複数のロールを割り当てることが可能である。ただし、以下の制約がある。

| # | 制約 | 説明 |
|---|------|------|
| 1 | 自己承認禁止 | CHANGE_REQとCAB_TECH/CAB_CHAIRを兼務する場合、自身が起票したRFCの承認はできない |
| 2 | 経営承認の独立性 | EXEC_APPROVERは他のロールとの兼務時でも、経営承認は独立した判断として記録される |
| 3 | 管理者制限 | SYS_ADMINは最大2名に制限し、他のロールとの兼務を非推奨とする |

## 5. 承認ワークフローのアクセス制御

### 5.1 変更種別ごとの承認要件

| 変更種別 | 必要承認者 | 最低承認数 | 自己承認 | セキュリティレビュー | 経営承認 |
|---------|-----------|:---------:|:------:|:-----------------:|:------:|
| 標準変更（Standard） | 自動承認 | - | - | 不要 | 不要 |
| 通常変更（Normal） | CAB議長 + 技術メンバー | 2名以上 | 禁止 | 条件付き | 不要 |
| 重大変更（Major） | CAB議長 + 技術メンバー + ビジネスメンバー + 経営承認者 | 3名以上 | 禁止 | 必須 | 必須 |
| 緊急変更（Emergency） | CAB議長 or 技術メンバー（ECAB） | 1名以上 | 禁止 | 事後 | 不要 |

### 5.2 職務分離マトリクス

以下の組み合わせは同一人物が同一RFCに対して行うことを禁止する。

| 操作A | 操作B | 分離要件 | 理由 |
|-------|-------|---------|------|
| RFC起票 | CAB承認 | 必須分離 | 自己承認防止 |
| RFC起票 | セキュリティレビュー | 推奨分離 | レビューの独立性確保 |
| CAB承認 | 変更実施 | 推奨分離 | チェック機能の確保 |
| RFC起票 | 経営承認 | 必須分離 | 承認の独立性確保 |

### 5.3 自己承認防止の実装

```
承認リクエスト受信
  │
  ├── 1. 承認者のユーザIDを取得
  │
  ├── 2. RFCの起票者（requester_id）を取得
  │
  ├── 3. 承認者 == 起票者 か判定
  │     │
  │     ├── YES → エラー: "自身が起票したRFCを承認することはできません"
  │     │         HTTP 403 Forbidden
  │     │
  │     └── NO → 承認処理を続行
  │
  └── 4. 判定結果を監査ログに記録（成功・拒否とも）
```

## 8. アクセス権の管理運用

### 8.1 アクセス権のライフサイクル

| フェーズ | 作業内容 | 責任者 | 頻度 |
|---------|---------|--------|------|
| 付与 | Entra IDグループへの追加、ロール割当て | IT部門長 | 都度（入社・異動時） |
| レビュー | 全ユーザのロール割当ての妥当性確認 | IT部門長 + セキュリティレビュア | 四半期 |
| 変更 | ロールの追加・削除、権限の変更 | IT部門長（SYS_ADMIN承認） | 都度 |
| 削除 | 退職者・異動者のアクセス権剥奪 | IT部門管理者 | 都度（退職・異動時） |
| 監査 | アクセス権の棚卸し、不要権限の特定 | セキュリティレビュア | 半期 |

### 8.2 緊急アクセス（ブレークグラス）

| 項目 | 内容 |
|------|------|
| 目的 | 通常の認証・認可が利用できない緊急事態への対応 |
| 対象 | SYS_ADMIN相当の緊急管理者アカウント（2つ） |
| 保管場所 | 物理的な金庫（IT部門長管理） |
| 使用条件 | Entra ID障害時、主要管理者不在時など |
| 使用手順 | 1. 緊急事態の宣言 → 2. 金庫から認証情報取得 → 3. ログインと操作 → 4. 操作完了後のパスワード変更 → 5. 使用報告書の作成 |
| 事後対応 | 全操作の監査ログレビュー、パスワード変更、使用報告書のCABレビュー |

### 8.3 特権アクセス管理

| 対策 | 実装内容 |
|------|----------|
| 特権アカウント制限 | SYS_ADMINは最大2名、専用アカウント使用 |
| 特権操作ログ | SYS_ADMINの全操作を強化ログ記録 |
| 特権セッション監視 | SYS_ADMINセッションの操作内容をリアルタイム記録 |
| 定期的な棚卸し | SYS_ADMIN権限の妥当性を四半期ごとにレビュー |
| JIT（Just-In-Time）アクセス | 通常は一般権限、管理作業時のみ一時的にSYS_ADMIN昇格（Entra ID PIM） |
