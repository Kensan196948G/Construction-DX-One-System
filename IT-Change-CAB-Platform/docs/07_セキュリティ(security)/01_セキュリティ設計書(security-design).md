## 1. セキュリティ設計方針

### 1.1 基本方針

本プラットフォームのセキュリティ設計は、以下の原則に基づく。

| # | 原則 | 説明 |
|---|------|------|
| 1 | 多層防御（Defense in Depth） | ネットワーク・アプリケーション・データの各層で独立した防御策を実装する |
| 2 | 最小権限の原則（Least Privilege） | 各ユーザ・サービスに業務上必要な最小限の権限のみを付与する |
| 3 | ゼロトラスト | すべてのアクセスを検証し、暗黙の信頼を排除する |
| 4 | セキュリティ・バイ・デザイン | 設計段階からセキュリティを組み込む |
| 5 | 説明責任（Accountability） | すべての操作を記録し、追跡可能とする |

### 1.2 準拠規格

| 規格 | 対象条項 | 本設計書での対応 |
|------|----------|------------------|
| ISO27001:2022 | A.8.32 変更管理 | 変更プロセス全体のセキュリティ制御 |
| NIST CSF 2.0 | PROTECT PR.IP | 情報保護プロセスの実装 |
| ISO20000-1:2018 | 変更管理プロセス | セキュリティを考慮した変更管理 |

## 2. 認証設計

### 2.1 認証方式

| 項目 | 設計内容 |
|------|----------|
| 認証基盤 | Microsoft Entra ID（旧Azure AD）によるSSO |
| 認証プロトコル | OpenID Connect（OIDC） / OAuth 2.0 Authorization Code Flow with PKCE |
| 多要素認証（MFA） | HENGEONE経由のMFAを必須化（全ユーザ） |
| セッション管理 | JWTベースのアクセストークン + リフレッシュトークン |
| トークン有効期限 | アクセストークン: 15分 / リフレッシュトークン: 8時間 |
| 外部API認証 | サービス間通信にはOAuth 2.0 Client Credentials Grantを使用 |

### 2.2 Entra ID連携構成

```
ユーザ
  │
  │ ブラウザアクセス
  ▼
HENGEONE（MFA認証）
  │
  │ MFA完了
  ▼
Microsoft Entra ID
  │
  │ OIDC認証フロー
  ▼
IT-Change-CAB-Platform（フロントエンド）
  │
  │ アクセストークン付与
  ▼
バックエンドAPI（トークン検証）
```

### 2.3 認証フロー詳細

| ステップ | 処理内容 | セキュリティ対策 |
|----------|----------|------------------|
| 1 | ユーザがフロントエンドにアクセス | HTTPS強制（HSTS有効） |
| 2 | HENGEONE経由でMFA認証 | TOTP/プッシュ通知による多要素認証 |
| 3 | Entra IDへのOIDCリダイレクト | state/nonceパラメータによるCSRF防止 |
| 4 | 認可コード取得 | PKCE（Proof Key for Code Exchange）使用 |
| 5 | バックエンドで認可コードをトークン交換 | サーバサイドでのトークン交換（クライアントシークレット保護） |
| 6 | JWTアクセストークン発行 | RS256署名、issuer/audience検証 |
| 7 | APIリクエスト | Bearerトークンによる認証 |

### 2.4 セッション管理

| 項目 | 設定値 | 説明 |
|------|--------|------|
| アクセストークン有効期限 | 15分 | 短期間で失効し再取得を強制 |
| リフレッシュトークン有効期限 | 8時間 | 業務時間内での再認証不要 |
| 絶対セッションタイムアウト | 12時間 | リフレッシュ不可の最大時間 |
| アイドルタイムアウト | 30分 | 操作なし時の自動ログアウト |
| 同時セッション数 | 最大3 | 同一ユーザの並行セッション制限 |
| トークン格納場所 | HttpOnly Secure Cookie | XSSによるトークン窃取を防止 |

## 4. 通信暗号化設計

### 4.1 通信経路暗号化

| 通信経路 | プロトコル | 暗号化方式 | 証明書管理 |
|----------|-----------|-----------|-----------|
| クライアント ←→ フロントエンド | HTTPS (TLS 1.3) | AES-256-GCM | Let's Encrypt / Azure App Service証明書 |
| フロントエンド ←→ バックエンドAPI | HTTPS (TLS 1.3) | AES-256-GCM | 内部CA証明書 |
| バックエンド ←→ PostgreSQL | TLS 1.3 | AES-256-GCM | サーバ証明書検証有効 |
| バックエンド ←→ Redis | TLS 1.3 | AES-256-GCM | Redis TLS設定 |
| バックエンド ←→ Entra ID | HTTPS (TLS 1.3) | AES-256-GCM | Microsoft管理証明書 |
| バックエンド ←→ Teams Webhook | HTTPS (TLS 1.3) | AES-256-GCM | Microsoft管理証明書 |
| バックエンド ←→ ITSM連携 | HTTPS (TLS 1.2以上) | AES-256-GCM | 相互TLS認証（mTLS） |

### 4.2 TLS設定

| 項目 | 設定 |
|------|------|
| 最小TLSバージョン | TLS 1.2（TLS 1.3推奨） |
| 暗号スイート（TLS 1.3） | TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256 |
| 暗号スイート（TLS 1.2） | ECDHE-RSA-AES256-GCM-SHA384, ECDHE-RSA-AES128-GCM-SHA256 |
| 無効化対象 | SSLv2, SSLv3, TLS 1.0, TLS 1.1, RC4, 3DES, NULL暗号 |
| HSTS | max-age=31536000; includeSubDomains; preload |
| 証明書ピンニング | API間通信のみ適用 |

## 5. データ保護設計

### 5.1 データ分類

| 分類 | 説明 | 該当データ | 保護レベル |
|------|------|-----------|-----------|
| 機密（Confidential） | 漏洩時に重大な影響 | 認証情報、APIキー、暗号鍵 | 暗号化必須・アクセスログ必須 |
| 社外秘（Internal） | 組織内限定情報 | RFC詳細、変更計画、影響分析結果 | アクセス制御・ログ記録 |
| 部門内（Restricted） | 特定部門限定 | CAB審議内容、投票記録 | ロールベースアクセス制御 |
| 一般（Public） | 公開可能情報 | 変更カレンダー（概要のみ）、KPIサマリ | 基本的なアクセス制御 |

### 5.2 保存データの暗号化

| データ種別 | 暗号化方式 | 鍵管理 |
|-----------|-----------|--------|
| データベース全体 | PostgreSQL TDE（透過的データ暗号化） | Azure Key Vault / HashiCorp Vault |
| 機密フィールド（APIキー等） | AES-256-GCM（アプリケーション層暗号化） | Azure Key Vault |
| バックアップデータ | AES-256暗号化 | バックアップ専用暗号鍵 |
| Redis キャッシュ | 機密データはキャッシュしない | - |
| ログデータ | 個人情報マスキング後に保存 | - |

### 5.3 データライフサイクル管理

| データ | 保存期間 | 保存期間後の処理 | 根拠 |
|--------|---------|------------------|------|
| RFC・変更記録 | 7年 | アーカイブ（暗号化済み） | ISO20000監査要件 |
| CAB議事録・承認記録 | 7年 | アーカイブ（暗号化済み） | ISO20000監査要件 |
| 監査ログ | 5年 | セキュアデリート | ISO27001要件 |
| セッション情報 | 24時間 | 自動削除 | セキュリティポリシー |
| 一時ファイル | 処理完了後即時 | セキュアデリート | セキュリティポリシー |

### 5.4 個人情報保護

| 対策 | 実装方法 |
|------|----------|
| 個人情報の最小化 | 業務に必要な最小限の個人情報のみ保持（氏名・メール・所属部門） |
| ログ内個人情報マスキング | ユーザIDをハッシュ化してログ記録 |
| データエクスポート制限 | CSVエクスポートはSYS_ADMINロールに限定 |
| データ削除 | ユーザ退職時のアカウント無効化・個人情報匿名化 |

### 6.3 ログ保存・管理

| 項目 | 設計内容 |
|------|----------|
| ログ保存先 | PostgreSQL audit_logsテーブル + 外部ログ基盤（Azure Monitor / ELK Stack） |
| ログローテーション | 日次ローテーション、90日間ホットストレージ |
| 長期保存 | 5年間コールドストレージ（Azure Blob Storage、暗号化済み） |
| ログ改ざん防止 | 追記専用テーブル（DELETE/UPDATE権限なし）、ハッシュチェーン |
| ログ検索 | Elasticsearch / Azure Log Analyticsによる全文検索 |
| アラート | 不正アクセス試行、権限昇格、異常パターンの自動アラート |

### 6.4 ログ改ざん防止設計

```
ログエントリ生成
  │
  ├── 1. タイムスタンプ付与（NTPサーバ同期）
  │
  ├── 2. 前回ログのハッシュ値を含めてチェーン化
  │     hash(n) = SHA-256(log_entry(n) + hash(n-1))
  │
  ├── 3. 追記専用テーブルに書き込み（DELETE/UPDATE権限なし）
  │
  └── 4. 非同期で外部ログ基盤に転送（改ざん検知用）
```

## 7. アプリケーションセキュリティ

### 7.1 入力値検証

| 対象 | バリデーション内容 |
|------|-------------------|
| RFC番号 | 正規表現 `/^RFC-\d{4}-\d{3,5}$/` |
| テキストフィールド | XSSサニタイズ（DOMPurify）、SQLインジェクション対策（パラメータ化クエリ） |
| 日時フィールド | ISO 8601形式チェック、未来日チェック |
| ファイルアップロード | MIMEタイプ検証、ファイルサイズ制限（10MB）、ウイルススキャン |
| URLパラメータ | パストラバーサル防止、ホワイトリスト検証 |
| JSONペイロード | JSONスキーマバリデーション（Ajv） |

### 7.2 OWASP Top 10対策

| # | 脅威 | 対策 |
|---|------|------|
| A01 | アクセス制御の不備 | RBAC実装、APIレベルの認可チェック、オブジェクトレベル認可（IDOR防止） |
| A02 | 暗号化の失敗 | TLS 1.3強制、機密データ暗号化、鍵管理にKey Vault使用 |
| A03 | インジェクション | パラメータ化クエリ（TypeORM）、入力値サニタイズ、CSP設定 |
| A04 | 安全でない設計 | 脅威モデリング実施、セキュリティレビュー必須化 |
| A05 | セキュリティの設定ミス | セキュリティヘッダ設定、不要サービス無効化、デフォルト認証情報の排除 |
| A06 | 脆弱なコンポーネント | Dependabot有効化、定期的な依存パッケージ更新、SCA実施 |
| A07 | 識別と認証の失敗 | Entra ID + HENGEONE MFA、アカウントロックアウト、パスワードポリシー |
| A08 | ソフトウェアとデータの整合性 | CI/CDパイプラインのセキュリティ、コード署名、依存関係の整合性チェック |
| A09 | セキュリティログと監視の不足 | 包括的監査ログ、リアルタイム監視、アラート設定 |
| A10 | SSRF | 外部URLアクセスのホワイトリスト制限、内部ネットワークへのリクエスト遮断 |

## 8. インフラストラクチャセキュリティ

### 8.1 ネットワークセキュリティ

| レイヤー | セキュリティ対策 |
|----------|------------------|
| 外部境界 | WAF（Azure Front Door / Cloudflare）、DDoS防御 |
| ネットワーク分離 | Docker Network分離、バックエンドのみがDBに接続可能 |
| 内部通信 | サービス間通信はTLS暗号化 |
| ポート制限 | 外部公開: 443のみ、内部通信: 必要最小限のポート |
| DNS | DNSSEC有効化 |

### 8.2 コンテナセキュリティ

| 対策 | 実装内容 |
|------|----------|
| ベースイメージ | Alpine Linux（最小構成）、定期的なイメージ更新 |
| 非rootユーザ | コンテナ内プロセスは非root（UID: 1000）で実行 |
| 読み取り専用ファイルシステム | アプリケーションコンテナのファイルシステムを読み取り専用化 |
| リソース制限 | CPU/メモリの上限設定（OOM防止） |
| イメージスキャン | Trivy / Snykによるコンテナイメージの脆弱性スキャン |
| シークレット管理 | Docker Secrets / Azure Key Vault（環境変数にシークレットを直接設定しない） |

### 8.3 データベースセキュリティ

| 対策 | 実装内容 |
|------|----------|
| 接続制限 | バックエンドコンテナからのみ接続可能（ネットワーク分離） |
| 認証 | 専用データベースユーザ、強力なパスワード（32文字以上ランダム） |
| 権限分離 | アプリケーション用（CRUD）、管理用（DDL）、読み取り専用の3ユーザ |
| 監査ログ | pgAuditによるSQL文の記録 |
| バックアップ暗号化 | AES-256によるバックアップの暗号化 |
| 接続暗号化 | TLS 1.3によるクライアント-サーバ間暗号化 |

## 9. セキュリティレビュー要件

### 9.1 変更申請時のセキュリティレビュー

以下の条件に該当する変更申請は、セキュリティレビュアによる審査を必須とする。

| # | 条件 | セキュリティレビュー要否 |
|---|------|-------------------------|
| 1 | 認証・認可に関する変更 | 必須 |
| 2 | 暗号化設定の変更 | 必須 |
| 3 | ファイアウォール・ネットワーク設定変更 | 必須 |
| 4 | 外部連携の新規追加・変更 | 必須 |
| 5 | データベーススキーマ変更（個人情報関連） | 必須 |
| 6 | Active Directory / Entra IDの変更 | 必須 |
| 7 | HENGEONE設定変更 | 必須 |
| 8 | 重大変更（Major） | 必須 |
| 9 | 標準変更（定型パッチ適用等） | 不要 |
| 10 | 上記以外の通常変更 | 影響分析結果に基づき判定 |

### 9.2 セキュリティレビュー観点

| # | レビュー観点 | 確認内容 |
|---|-------------|----------|
| 1 | 認証への影響 | 認証フローの変更有無、MFA要件の維持 |
| 2 | 認可への影響 | 権限変更、最小権限原則の遵守 |
| 3 | データ保護 | 機密データの取り扱い、暗号化要件 |
| 4 | 通信セキュリティ | 暗号化プロトコル、証明書管理 |
| 5 | 監査証跡 | ログ記録の継続性、改ざん防止 |
| 6 | コンプライアンス | ISO27001/ISO20000/NIST CSF要件への影響 |
| 7 | 脆弱性 | 既知の脆弱性、セキュリティパッチの適用状況 |
| 8 | ロールバック | セキュリティ設定のロールバック可否 |

## 11. インシデント対応

### 11.1 セキュリティインシデント検知

| 検知対象 | 検知方法 | アラート先 |
|----------|----------|-----------|
| 認証失敗の連続（5回以上/5分） | ログ分析 | セキュリティレビュア、SYS_ADMIN |
| 権限昇格の試行 | APIアクセスログ | セキュリティレビュア、CAB議長 |
| 異常なデータアクセスパターン | 行動分析 | セキュリティレビュア |
| 未認可のAPI呼び出し | APIゲートウェイログ | SYS_ADMIN |
| 既知の攻撃パターン | WAFログ | SYS_ADMIN |

### 11.2 対応フロー

```
セキュリティイベント検知
  │
  ├── 1. 自動アラート送信（Teams / メール）
  │
  ├── 2. インシデント分類（重大度判定）
  │     ├── Critical: アカウント侵害、データ漏洩
  │     ├── High: 認証突破試行、権限昇格
  │     ├── Medium: 異常アクセスパターン
  │     └── Low: ポリシー違反（軽微）
  │
  ├── 3. 初動対応
  │     ├── Critical: 即座にアカウントロック、ネットワーク遮断
  │     ├── High: 対象アカウント監視強化
  │     └── Medium/Low: ログ分析、原因調査
  │
  └── 4. 事後対応
        ├── インシデントレポート作成
        ├── 再発防止策の策定
        └── セキュリティポリシー見直し
```

## 12. セキュリティテスト

### 12.1 テスト種別

| テスト種別 | 実施頻度 | 実施者 | 内容 |
|-----------|---------|--------|------|
| 静的アプリケーションセキュリティテスト（SAST） | CI/CDパイプライン毎 | 自動（SonarQube / CodeQL） | ソースコードの脆弱性検出 |

| コンテナイメージスキャン | ビルド時 | 自動（Trivy） | コンテナイメージの脆弱性検出 |
| ペネトレーションテスト | 年次 | 外部セキュリティベンダー | システム全体の侵入テスト |
| セキュリティ設定レビュー | 四半期 | セキュリティレビュア | インフラ・アプリの設定確認 |
