## 1. ワークフロー概要

### 1.1 変更管理プロセス全体フロー

```
RFC起票 → 提出 → 分類 → CAB審議 → 承認 → 実施 → 完了 → PIR → クローズ
  │                        │         │       │       │
  │                     Emergency    │    ロールバック │
  │                     →ECAB承認    │    →PIR必須    │
  │                                  │               │
  └── 下書き保存                 Standard           トレンド
      取り消し                   →自動承認           分析・KPI
```

### 1.2 変更種別ごとのワークフロー比較

| フェーズ | Standard | Normal | Major | Emergency |
|---------|----------|--------|-------|-----------|
| 起票 | 標準フォーム | 全項目入力 | 全項目入力 | 簡易フォーム |
| 影響分析 | テンプレート適用 | 手動+自動分析 | 手動+自動分析 | 事後記録可 |
| ロールバック計画 | 任意 | 必須 | 必須 | 事後記録可 |
| 承認 | 自動承認 | CAB承認(2名+議長) | CAB+経営承認(3名+議長+経営) | ECAB承認(1名) |
| 事前通知 | 不要 | 5営業日前 | 10営業日前 | 事後報告 |
| 承認期限 | 即時 | 5営業日 | 10営業日 | 4時間 |
| 実施チェック | 簡易 | 標準 | 厳格 | 簡易 |
| PIR | 任意 | 失敗時必須 | 必須 | 必須 |
| 事後CAB報告 | 不要 | 不要 | 必要 | 必須 |

## 2. Standard変更（標準変更）ワークフロー

### 2.1 フロー図

```
┌────────┐    ┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ 起票    │──>│ 提出      │──>│ 自動承認      │──>│ 実施開始      │
│ (draft) │   │(submitted)│   │(auto_approved)│   │(in_progress) │
└────────┘    └──────────┘    └──────────────┘    └───┬──────────┘
                                                      │
                                      ┌───────────────┤
                                      │               │
                                      ▼               ▼
                                ┌──────────┐    ┌──────────────┐
                                │ 完了      │    │ ロールバック   │
                                │(completed)│    │(rolled_back) │
                                └──────────┘    └──────────────┘
```

### 2.2 詳細ステップ

| ステップ | 処理内容 | 条件・バリデーション |
|---------|---------|-------------------|
| 1. 起票 | 標準変更テンプレートから起票 | テンプレートマッチング |
| 2. 提出 | 必須項目バリデーション | 基本項目のみ必須 |
| 3. 自動承認 | システムが自動承認 | テンプレート一致確認 |
| 4. 実施 | 簡易チェックリストで実施 | バックアップ確認のみ |
| 5. 完了/ロールバック | 結果記録 | 実施結果の記録必須 |

### 2.3 標準変更テンプレート例

| テンプレートID | テンプレート名 | 対象システム | 自動承認条件 |

| STD-002 | パスワードリセット | AD, EntraID | 影響ユーザ1名以下 |

### 3.2 承認ワークフロー詳細

| ステップ | 処理 | SLA |
|---------|------|-----|
| 1. 提出 | RFCがsubmitted状態に遷移。自動で変更種別判定 | 即時 |
| 2. 議題登録 | 次回CAB会議の議題に自動登録 | 即時 |
| 3. 通知送信 | CABメンバーに承認依頼通知（メール+Teams） | 1分以内 |
| 4. 事前レビュー | CABメンバーが事前にRFC内容を確認 | 会議前 |
| 5. CAB審議 | 会議内またはオンラインで審議・投票 | 5営業日以内 |
| 6. 承認判定 | 2名以上 + CAB議長の承認で承認確定 | 即時 |
| 7. 結果通知 | 申請者に承認/却下結果を通知 | 1分以内 |

### 3.3 承認判定ロジック

```
function judgeApproval(votes, changeType):
    config = getApprovalConfig(changeType)

    // 必須承認者チェック
    for requiredApprover in config.requiredApprovers:
        if not hasApproved(requiredApprover, votes):
            return PENDING

    // 拒否権チェック
    for vetoHolder in config.vetoHolders:
        if hasRejected(vetoHolder, votes):
            return REJECTED

    // 承認数チェック
    approvalCount = countApprovals(votes)
    if approvalCount >= config.minimumApprovals:
        return APPROVED

    // 全員投票済みで承認数不足
    if allVoted(votes):
        return REJECTED

    return PENDING
```

## 4. Major変更（重大変更）ワークフロー

### 4.1 フロー図

```
┌────────┐    ┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ 起票    │──>│ 提出      │──>│ CAB審議       │──>│ 経営承認      │
│ (draft) │   │(submitted)│   │(cab_review)  │   │(exec_review) │
└────────┘    └──────────┘    └──────────────┘    └───┬──────────┘
                                                      │
                                      ┌───────────────┤
                                      │               │
                                      ▼               ▼
                                ┌──────────┐    ┌──────────┐
                                │ 承認      │    │ 却下      │
                                │(approved) │    │(rejected) │
                                └────┬─────┘    └──────────┘
                                     │
                                     ▼
                               ┌──────────────┐
                               │ 実施開始      │
                               │(in_progress) │
                               └───┬──────┬───┘
                                   │      │
                              完了  │      │ ロールバック
                                   ▼      ▼
                              ┌─────┐  ┌──────────┐
                              │完了  │  │ロールバック│
                              └──┬──┘  └────┬─────┘
                                 │          │
                                 ▼          ▼
                              ┌──────────────┐
                              │ PIR（必須）    │
                              │(pir_pending) │
                              └──────┬───────┘
                                     │
                                     ▼ CAB事後報告
                              ┌──────────┐
                              │ クローズ   │
                              │(closed)   │
                              └──────────┘
```

### 4.2 Major変更固有の要件

| 要件 | 内容 |
|------|------|
| 事前通知 | 10営業日前にCABメンバー・経営層に通知 |
| 承認者 | CABメンバー3名以上 + CAB議長 + 経営承認者 |
| 承認期限 | 10営業日以内 |
| セキュリティレビュー | コアシステム変更時は必須 |
| ロールバック計画 | 詳細手順・検証手順・連絡先一覧が必須 |
| 実施チェックリスト | 厳格チェック（テスト環境検証必須） |
| PIR | 成否に関わらず必須 |
| CAB事後報告 | 次回CAB会議で実施結果を報告 |

## 5. Emergency変更（緊急変更）ワークフロー

### 5.1 フロー図

```
┌────────────┐    ┌──────────────┐    ┌──────────────┐
│ 緊急起票    │──>│ ECAB通知      │──>│ ECAB承認      │
│(簡易フォーム)│   │(ecab_pending)│   │(1名で承認可)  │
└────────────┘    └──────────────┘    └───┬──────────┘
                                          │
                                          ▼
                                    ┌──────────────┐
                                    │ 実施開始      │
                                    │(in_progress) │
                                    └───┬──────┬───┘
                                        │      │
                                   完了  │      │ ロールバック
                                        ▼      ▼
                                   ┌─────┐  ┌──────────┐
                                   │完了  │  │ロールバック│
                                   └──┬──┘  └────┬─────┘
                                      │          │
                                      ▼          ▼
                                   ┌──────────────┐
                                   │ 事後記録      │
                                   │(影響分析・    │
                                   │ ロールバック   │
                                   │ 計画の事後入力)│
                                   └──────┬───────┘
                                          │
                                          ▼
                                   ┌──────────────┐
                                   │ PIR（必須）    │
                                   │(pir_pending) │
                                   └──────┬───────┘
                                          │
                                          ▼ 次回CABで事後報告・承認
                                   ┌──────────────┐
                                   │ CAB事後承認   │
                                   │(post_cab)    │
                                   └──────┬───────┘
                                          │
                                          ▼
                                   ┌──────────┐
                                   │ クローズ   │
                                   │(closed)   │
                                   └──────────┘
```

### 5.2 ECAB承認フロー詳細

| ステップ | 処理 | SLA |
|---------|------|-----|
| 1. 緊急起票 | 簡易フォームで最低限の情報入力 | 即時 |
| 2. 緊急通知 | CAB議長・ECABメンバーにTeams+メール送信 | 5分以内 |
| 3. ECAB審議 | オンラインで緊急審議 | 30分以内 |
| 4. 承認判定 | CAB議長1名の承認で実施可 | 4時間以内 |
| 5. 実施 | 簡易チェックで即時実施 | 承認後即時 |
| 6. 事後記録 | 影響分析・ロールバック計画の事後入力 | 1営業日以内 |
| 7. PIR | 事後レビュー実施 | 5営業日以内 |
| 8. CAB事後報告 | 次回定期CABで事後報告・承認 | 次回CAB |

### 5.3 ECAB代理承認

| 条件 | 代理承認者 |
|------|----------|
| CAB議長不在 | 技術メンバーのうちシニアメンバー |
| 全ECABメンバー不在 | IT部門長が指名した代理者 |
| 深夜・休日 | オンコール担当者 |

## 6. ロールバックワークフロー

### 6.1 ロールバック判断フロー

```
                    変更実施中 (in_progress)
                          │
                          ▼
                   ┌──────────────┐
                   │ロールバック     │
                   │判断基準チェック  │
                   └───┬──────┬───┘
                       │      │
              該当する   │      │ 該当しない
                       │      │
                       ▼      ▼
              ┌──────────┐  ┌──────────┐
              │ロールバック │  │実施継続    │
              │開始       │  │           │
              └────┬─────┘  └──────────┘
                   │
                   ▼
              ┌──────────────┐
              │関係者緊急通知  │
              │(Teams+メール) │
              └────┬─────────┘
                   │
                   ▼
              ┌──────────────┐
              │ロールバック     │
              │手順実行       │
              └────┬─────────┘
                   │
                   ▼
              ┌──────────────┐
              │正常復帰確認   │
              │              │
              └───┬──────┬───┘
                  │      │
             成功  │      │ 失敗
                  │      │
                  ▼      ▼
           ┌─────────┐ ┌─────────────┐
           │ロールバック│ │エスカレーション│
           │完了記録  │ │（上位者に報告）│
           └────┬────┘ └─────────────┘
                │
                ▼
           ┌──────────┐
           │PIR自動発行│
           └──────────┘
```

### 6.2 ロールバック判断基準例

| 判断基準 | 閾値 | 判断 |
|---------|------|------|
| サービス可用性 | 対象サービス停止 | 即座にロールバック |
| エラー率 | 変更前比5%以上増加 | ロールバック検討 |
| 応答時間 | 変更前比200%以上劣化 | ロールバック検討 |
| ユーザ影響 | 10名以上からの障害報告 | ロールバック検討 |
| 所要時間超過 | 計画時間の150%超過 | ロールバック判断 |

## 8. 承認エスカレーション

### 8.1 エスカレーションルール

| 条件 | エスカレーション先 | アクション |
|------|----------------|----------|
| Normal変更: 承認期限の50%経過 | CABメンバー | リマインダー通知 |
| Normal変更: 承認期限の80%経過 | CAB議長 | エスカレーション通知 |
| Normal変更: 承認期限超過 | CAB議長 | 自動保留（deferred）+ アラート |
| Major変更: 承認期限の50%経過 | CABメンバー + 経営承認者 | リマインダー通知 |
| Major変更: 承認期限超過 | IT部門長 | エスカレーション通知 |
| Emergency: 1時間未承認 | ECAB全メンバー | 再通知 |
| Emergency: 4時間未承認 | IT部門長 | 代理承認依頼 |

### 8.2 エスカレーションフロー

```
承認依頼送信
     │
     │ 期限50%経過
     ▼
リマインダー送信
     │
     │ 期限80%経過
     ▼
エスカレーション通知（上位者）
     │
     │ 期限100%超過
     ▼
自動保留 + 管理者アラート
```
