## 1. はじめに

### 1.1 本ガイドの対象者

本ガイドは、本プラットフォームのシステム管理者を対象としています。ユーザー管理、システム設定、フリーズ期間管理、承認ルール設定などの管理機能について説明します。

### 1.2 管理者権限

| 権限レベル | 操作可能範囲 |
|-----------|------------|
| システム管理者 | 全設定変更、ユーザー管理、システム設定 |
| CAB管理者 | CAB設定、承認ルール設定、フリーズ期間管理 |
| レポート管理者 | レポート設定、KPI設定 |

---

## 2. 管理画面へのアクセス

1. ログイン後、上部メニュー「管理」をクリック
2. 管理者権限を持つユーザーのみアクセス可能
3. 管理メニューが表示される

| メニュー | 機能概要 |
|---------|---------|
| ユーザー管理 | ユーザーの追加・編集・無効化、ロール割当 |
| システム設定 | システム全体の設定変更 |
| フリーズ期間管理 | 変更禁止期間の設定・編集・削除 |
| 承認ルール設定 | 承認条件・承認者の設定 |
| 通知設定 | 通知テンプレート・チャネルの設定 |
| 監査ログ | 操作履歴の確認 |
| システム状態 | ヘルスチェック・リソース状況 |

## 3. ユーザー管理

### 3.1 ユーザー一覧

1. 管理メニュー「ユーザー管理」をクリック
2. 登録ユーザーの一覧が表示される

| 表示項目 | 説明 |
|---------|------|
| ユーザーID | 一意のID |
| 氏名 | 表示名 |
| メールアドレス | ログインID兼通知先 |
| 部門 | 所属部門 |
| ロール | 割り当てられたロール（複数可） |
| ステータス | 有効/無効 |
| 最終ログイン | 最終ログイン日時 |

### 3.2 ユーザーの追加

1. 「ユーザー追加」ボタンをクリック
2. 以下の情報を入力

| 入力項目 | 必須 | 説明 |
|---------|------|------|
| 氏名 | 必須 | ユーザーの表示名 |
| メールアドレス | 必須 | ログインIDおよび通知先 |
| 部門 | 必須 | 所属部門を選択 |
| ロール | 必須 | 割り当てるロールを選択（複数可） |
| 初期パスワード | 必須 | 初回ログイン用パスワード |

3. 「登録」をクリック
4. ユーザーにアカウント作成通知メールが送信される

### 3.3 ロール定義

| ロール | 説明 | 主な権限 |
|-------|------|---------|
| user | 一般ユーザー | RFC起票、自分のRFC管理、カレンダー参照 |
| cab_member | CABメンバー | RFC審議、投票、CABダッシュボード参照 |
| cab_chair | CAB議長 | CABセッション管理、最終承認、ECAB承認 |
| security_reviewer | セキュリティレビュアー | セキュリティ関連RFCの審議 |
| admin | システム管理者 | 全管理機能 |
| report_viewer | レポート閲覧者 | KPIダッシュボード・レポート参照 |

### 3.4 ユーザーの編集

1. ユーザー一覧で対象ユーザーの「編集」ボタンをクリック
2. 情報を変更して「保存」をクリック

### 3.5 ユーザーの無効化

1. ユーザー一覧で対象ユーザーの「無効化」ボタンをクリック
2. 確認ダイアログで「無効化する」をクリック
3. 無効化されたユーザーはログイン不可となる
4. 過去のRFC・承認記録は保持される

## 4. フリーズ期間管理

### 4.1 フリーズ期間の設定

1. 管理メニュー「フリーズ期間管理」をクリック
2. 「新規フリーズ期間作成」をクリック
3. 以下の情報を入力

| 入力項目 | 必須 | 説明 | 入力例 |
|---------|------|------|--------|
| 期間名 | 必須 | フリーズ期間の名前 | 「2026年度末凍結期間」 |
| 開始日 | 必須 | フリーズ期間の開始日 | 2026-03-15 |
| 終了日 | 必須 | フリーズ期間の終了日 | 2026-03-31 |
| ブロック対象変更種別 | 必須 | ブロックする変更種別を選択 | 通常変更、重大変更 |
| 理由 | 任意 | フリーズ期間の理由 | 「年度末決算処理のため」 |

4. 「保存」をクリック

### 4.2 フリーズ期間の影響

| 設定 | 効果 |
|------|------|
| 通常変更をブロック | フリーズ期間中の通常変更RFC起票が警告付きでブロックされる |
| 重大変更をブロック | フリーズ期間中の重大変更RFC起票がブロックされる |
| 緊急変更は常に許可 | 緊急変更はフリーズ期間中でも起票・承認可能 |
| 標準変更は常に許可 | 標準変更（定型作業）はフリーズ期間中でも許可 |

### 4.3 推奨フリーズ期間設定

| 期間名 | 開始日 | 終了日 | ブロック対象 | 理由 |
|--------|--------|--------|------------|------|
| 年度末凍結 | 3月第3月曜日 | 3月末日 | 通常変更、重大変更 | 年度末決算 |
| 半期末凍結 | 9月第3月曜日 | 9月末日 | 通常変更、重大変更 | 半期末決算 |
| 年末年始凍結 | 12月25日 | 1月5日 | 通常変更、重大変更 | 年末年始休暇 |
| GW凍結 | 4月28日 | 5月6日 | 重大変更 | ゴールデンウィーク |

## 5. 承認ルール設定

### 5.1 承認ルールの管理

1. 管理メニュー「承認ルール設定」をクリック
2. 変更種別ごとの承認ルールが表示される

### 5.2 承認ルール設定項目

| 設定項目 | 説明 | デフォルト値 |
|---------|------|------------|
| 必要承認者数 | 承認に必要な投票数 | 通常変更: 2名、重大変更: 3名 |
| 承認可能ロール | 承認権限を持つロール | cab_member, cab_chair |
| 自動承認条件 | 標準変更の自動承認条件 | 変更種別が「標準変更」 |
| 承認期限 | 投票期限（営業日） | 5営業日 |
| エスカレーション期限 | 期限超過時のエスカレーション | 3営業日超過で議長に通知 |

### 5.3 承認ルールの変更手順

1. 対象の変更種別の「編集」をクリック
2. 設定値を変更
3. 「保存」をクリック
4. 変更は即時反映される（既に審議中のRFCには影響しない）

### 5.4 承認ルール変更の例

| シナリオ | 設定変更 |
|---------|---------|
| セキュリティ変更の承認を厳格化 | セキュリティ影響ありのRFCに対し、security_reviewer の承認を必須化 |
| 小規模変更の承認を簡略化 | 影響ユーザー10名以下の通常変更は承認者1名に変更 |
| ECAB承認者の追加 | 副議長ロールを追加し、ECAB承認可能者を増やす |

## 6. 通知設定

### 6.1 通知チャネル設定

1. 管理メニュー「通知設定」をクリック
2. 通知チャネルの設定を確認・変更

| 設定項目 | 説明 |
|---------|------|
| Teams Webhook URL | Microsoft Teams通知用のWebhook URL |
| SMTPサーバー | メール通知用のSMTPサーバー設定 |
| メール送信元 | 通知メールの送信元アドレス |

### 6.2 通知テンプレート編集

各通知種別のテンプレートを編集できます。

| 通知種別 | テンプレート変数 |
|---------|---------------|
| CAB審議依頼 | {rfcNumber}, {title}, {targetSystems}, {riskLevel}, {plannedStart} |
| RFC承認通知 | {rfcNumber}, {title}, {approverName}, {conditions} |
| RFC却下通知 | {rfcNumber}, {title}, {rejectReason} |
| フリーズ期間警告 | {rfcNumber}, {freezeName}, {freezeStart}, {freezeEnd} |
| 完了通知 | {rfcNumber}, {title}, {completedAt}, {implementer} |

## 7. 監査ログ

### 7.1 監査ログの確認

1. 管理メニュー「監査ログ」をクリック
2. 操作履歴が時系列で表示される

### 7.2 フィルタリング

| フィルタ | 説明 |
|---------|------|
| 期間 | 開始日～終了日 |
| ユーザー | 特定ユーザーの操作 |
| 操作種別 | ログイン/RFC操作/承認操作/設定変更 |
| 対象 | 対象リソース（RFC番号、ユーザーID等） |

### 7.3 監査ログの記録内容

| 項目 | 説明 |
|------|------|
| 日時 | 操作が行われた日時 |
| ユーザー | 操作を行ったユーザー |
| IPアドレス | アクセス元IPアドレス |
| 操作種別 | 実行された操作の種別 |
| 対象 | 操作対象のリソース |
| 詳細 | 操作の詳細（変更前後の値など） |
| 結果 | 成功/失敗 |

### 7.4 監査ログのエクスポート

1. 「エクスポート」ボタンをクリック
2. 期間を指定
3. CSV形式でダウンロード
4. ISO20000監査対応の証跡として利用可能

## 8. システム状態の確認

### 8.1 ヘルスチェック画面

1. 管理メニュー「システム状態」をクリック
2. 各サービスの稼働状況が表示される

| サービス | 表示内容 |
|---------|---------|
| バックエンドAPI | 稼働状態、応答時間、バージョン |
| PostgreSQL | 接続状態、接続数、DB サイズ |
| Redis | 接続状態、メモリ使用量 |
| BullMQ ワーカー | ジョブ処理状況、失敗ジョブ数 |
| フロントエンド | 稼働状態 |

### 8.2 リソースモニタリング

| メトリクス | 表示内容 |
|-----------|---------|
| CPU使用率 | 各コンテナのCPU使用率推移 |
| メモリ使用率 | 各コンテナのメモリ使用率推移 |
| ディスク使用率 | ストレージ使用量 |
| API応答時間 | エンドポイント別の応答時間推移 |

### 8.3 メンテナンスモード

1. 「メンテナンスモード」スイッチをクリック
2. メンテナンスメッセージを入力
3. 終了予定時刻を入力
4. 「有効化」をクリック
5. ユーザーにメンテナンス画面が表示される

## 9. データ管理

### 9.1 データエクスポート

| エクスポート対象 | 形式 | 用途 |
|---------------|------|------|
| RFC一覧 | CSV / Excel | 定期報告、分析 |

### 9.2 データ保持ポリシー

| データ種別 | 保持期間 | 根拠 |
|-----------|---------|------|
| RFC記録 | 5年 | ISO20000監査要件 |
| CAB議事録 | 5年 | ISO20000監査要件 |
| 監査ログ | 3年 | セキュリティポリシー |
| 通知履歴 | 1年 | 運用要件 |
| セッションログ | 90日 | 運用要件 |

## 10. トラブルシューティング（管理者向け）

| 問題 | 確認事項 | 対処法 |
|------|---------|--------|
| ユーザーがログインできない | アカウントの有効/無効状態を確認 | アカウント有効化、パスワードリセット |
| 通知が届かない | 通知設定のWebhook URL/SMTP設定を確認 | 設定修正、テスト通知送信 |
| 承認処理が動かない | 承認ルール設定を確認 | ルール設定の確認・修正 |
| システムが遅い | システム状態画面でリソースを確認 | リソース増強、不要データ削除 |
| 監査ログが表示されない | ログ保持期間を確認 | 期間設定の確認 |
