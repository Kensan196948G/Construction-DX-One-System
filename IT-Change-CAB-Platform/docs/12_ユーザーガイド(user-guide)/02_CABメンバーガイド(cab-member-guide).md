## 1. はじめに

### 1.1 本ガイドの対象者

本ガイドは、CAB（変更諮問委員会）メンバーとして変更申請（RFC）の審議・承認を行う方を対象としています。

### 1.2 CABメンバーの役割

| 役割 | 責務 |
|------|------|
| CAB 議長 | CAB会議の主催、最終承認判断、エスカレーション対応 |
| 技術メンバー | 技術的観点からのRFC評価、リスク分析 |
| ビジネスメンバー | 業務影響の観点からのRFC評価 |
| セキュリティレビュアー | セキュリティ影響の観点からのRFC評価 |

### 1.3 承認権限

| 変更種別 | 承認に必要な権限・人数 |
|---------|---------------------|
| 標準変更 | 自動承認（CAB審議不要） |
| 通常変更 | CABメンバー2名以上の承認 |
| 重大変更 | CABメンバー3名以上の承認 + 経営層承認 |
| 緊急変更 | CAB議長1名の承認（ECAB） |

## 2. CABダッシュボード

### 2.1 ダッシュボードへのアクセス

1. ログイン後、上部メニュー「CABダッシュボード」をクリック
2. CABメンバー権限を持つユーザーのみアクセス可能

### 2.2 ダッシュボード画面構成

| エリア | 表示内容 |
|--------|---------|
| 審議待ちRFC | 未審議のRFC一覧（件数バッジ付き） |
| 次回CABセッション | 次回のCAB会議日時、議題数 |
| 最近の審議結果 | 直近の承認/却下結果一覧 |
| CAB統計 | 今月の審議件数、承認率、平均リードタイム |
| 自分の投票状況 | 自分がまだ投票していないRFC一覧 |

## 3. RFC審議手順

### 3.1 審議対象RFCの確認

1. CABダッシュボードの「審議待ちRFC」一覧を確認
2. 対象RFCをクリックして詳細画面を開く

### 3.2 RFC評価のポイント

RFCを審議する際、以下の観点で評価してください。

| 評価観点 | 確認ポイント |
|---------|------------|
| 変更の必要性 | 変更の目的が明確か、ビジネス上の正当性があるか |
| 影響分析の妥当性 | 影響範囲が適切に評価されているか、見落としはないか |
| リスク評価 | 技術リスクの評価は適切か、対策は十分か |
| 実施計画 | 日程は妥当か、作業時間は十分か |
| ロールバック計画 | 切り戻し手順は具体的か、実行可能か |
| テスト計画 | 変更の検証方法は適切か |
| セキュリティ | セキュリティへの影響はないか |
| フリーズ期間 | フリーズ期間との衝突はないか |
| 衝突 | 他の変更との衝突はないか |

### 3.3 審議画面の操作

RFC詳細画面の「審議」タブで以下の操作が可能です。

#### コメント追加

1. 「コメント」欄に質問・指摘事項を入力
2. 「コメント投稿」ボタンをクリック
3. 申請者に通知が送信される

#### 投票

1. RFC内容を十分に確認した上で投票ボタンを選択

| ボタン | 効果 | 説明 |
|--------|------|------|
| 承認 | 当該RFCを承認する投票 | 変更内容に問題なしと判断 |
| 却下 | 当該RFCを却下する投票 | 理由の記入が必須 |
| 追加情報要求 | 追加情報を要求 | 要求内容の記入が必須 |
| 延期 | 次回CABに延期 | 延期理由の記入推奨 |

2. 必要に応じてコメントを入力
3. 「投票確定」ボタンをクリック

**注意:** 一度投票すると変更できません。慎重に判断してください。

## 4. 承認・却下操作の詳細

### 4.1 承認操作

1. RFC詳細画面で内容を確認
2. 「承認」ボタンをクリック
3. 承認条件がある場合は「条件」欄に記入（例：「本番デプロイ前にステージング環境でのテスト結果を報告すること」）
4. 「投票確定」をクリック
5. 必要承認数に到達すると、RFCステータスが自動的に「承認済み」に変更される
6. 申請者に承認通知が自動送信される

### 4.2 却下操作

1. RFC詳細画面で内容を確認
2. 「却下」ボタンをクリック
3. **却下理由を必ず記入**（例：「ロールバック計画が不十分です。具体的な切り戻し手順を追記してください。」）
4. 「投票確定」をクリック
5. RFCステータスが「却下」に変更される
6. 申請者に却下理由付きの通知が自動送信される

### 4.3 条件付き承認

1. 「承認」ボタンをクリック
2. 「承認条件」欄に条件を記入
3. 条件を満たした後に実施可能となる

| 条件例 | 用途 |
|--------|------|
| 「テスト結果報告後に実施可」 | テスト完了を確認してからの実施 |
| 「実施前に関係部門へ周知すること」 | 周知確認の必要がある場合 |
| 「ネットワーク担当との同席作業とすること」 | 特定メンバーの立ち会いが必要な場合 |

## 5. ECAB（緊急CAB）対応

### 5.1 緊急変更の承認フロー

緊急変更が申請された場合、CAB議長に即時通知されます。

1. Teams/メールで緊急承認依頼通知を受信
2. 通知内のリンクからRFC詳細画面を開く
3. 緊急変更の内容を確認
4. 「緊急承認」ボタンをクリック
5. 承認理由を記入
6. 「確定」をクリック

### 5.2 緊急変更の確認ポイント

| 確認項目 | 内容 |
|---------|------|
| 緊急性の妥当性 | 本当に緊急対応が必要か |
| 関連インシデント | 関連するインシデントが存在するか |
| 最小限の変更範囲 | 必要最小限の変更に限定されているか |
| ロールバック手段 | 最低限の切り戻し手段が確保されているか |

## 6. CABセッション管理（議長向け）

### 6.1 CABセッションの作成

1. CABダッシュボードの「セッション管理」を開く
2. 「新規セッション作成」をクリック
3. 以下の情報を入力

| 入力項目 | 説明 |
|---------|------|
| 開催日時 | CAB会議の日時 |
| 参加者 | CABメンバーを選択 |
| 議題自動生成 | ONにすると審議待ちRFCが自動追加される |
| 追加議題 | その他の議題があれば記入 |

4. 「作成」をクリック
5. 参加者に召集通知が自動送信される

### 6.2 定期CABの設定

1. 「定期スケジュール設定」をクリック
2. 開催曜日・時刻を設定（例：毎週水曜日 14:00）
3. 参加者を設定
4. 「保存」をクリック

### 6.3 セッションの実施

1. CABセッション画面を開く
2. 議題のRFCを順番に審議
3. 各メンバーの投票を確認
4. 議長として最終判定を記入
5. 全議題の審議完了後、「セッション終了」をクリック
6. 議事録が自動生成される

## 7. 議事録の確認

### 7.1 議事録の確認手順

1. CABダッシュボードの「議事録一覧」をクリック
2. 対象の議事録を選択
3. 議事録の内容を確認

### 7.2 議事録の内容

| セクション | 内容 |
|-----------|------|
| 会議情報 | 開催日時、参加者一覧 |
| 審議結果一覧 | 各RFCの審議結果（承認/却下/延期）のサマリー |
| RFC別詳細 | 各RFCの投票内訳、コメント、承認条件 |
| その他議題 | RFC以外の議題の記録 |

### 7.3 議事録のエクスポート

1. 議事録画面の「PDFエクスポート」ボタンをクリック
2. PDF形式の議事録がダウンロードされる
3. ISO20000監査用の証跡として利用可能

## 8. CABメンバーのベストプラクティス

| 項目 | 推奨事項 |
|------|---------|
| 事前確認 | CABセッション前に審議対象RFCを事前確認する |
| タイムリーな投票 | オンライン承認の場合は3営業日以内に投票する |
| 具体的なフィードバック | 却下・差し戻し時は具体的な理由と改善提案を記入する |
| リスクの多角的評価 | 技術面だけでなく、業務影響・セキュリティ面からも評価する |
| 利益相反の申告 | 自分が申請者のRFCの審議時は利益相反を申告する |
