## 1. テスト計画概要

### 1.1 目的

本テスト計画書は、IT変更管理・リリース自動化プラットフォームの品質を保証するために、全テストフェーズの戦略・方針・スケジュールを定義する。ISO20000準拠の変更管理プロセスが正しく動作することを検証し、非機能要件（性能・可用性・セキュリティ）を満たすことを確認する。

### 1.2 テスト対象範囲

| 対象コンポーネント | 技術スタック | テスト対象 |
|-------------------|-------------|-----------|
| フロントエンド | React 18 / TypeScript / Ant Design | RFC管理UI、CABダッシュボード、変更カレンダー、KPIダッシュボード |
| バックエンドAPI | Node.js / Express / TypeScript | RFC管理API、CAB承認API、変更実施API、PIRレビューAPI |
| データベース | PostgreSQL 16 | テーブル定義、データ整合性、マイグレーション |
| キャッシュ/キュー | Redis 7 / BullMQ | キャッシュ動作、ジョブキュー処理 |
| 外部連携 | Teams Webhook / メール / GitHub Actions | 通知送信、CI/CDパイプライン連携 |

### 1.3 テスト対象外

- 外部システム（Active Directory、Entra ID、HENGEONE等）の内部動作
- ブラウザ固有の描画エンジンテスト（対応ブラウザでの基本動作確認は実施）
- 物理インフラ層のテスト

## 2. テスト戦略

### 2.1 テストレベル

| テストレベル | 目的 | 実施者 | 自動化率目標 |
|-------------|------|--------|-------------|
| 単体テスト（Unit Test） | 個々の関数・コンポーネントの正確性検証 | 開発者 | 90%以上 |
| 結合テスト（Integration Test） | コンポーネント間の連携動作検証 | 開発者 | 80%以上 |
| E2Eテスト（End-to-End Test） | ユーザーシナリオベースの全体動作検証 | QAチーム | 70%以上 |
| 性能テスト（Performance Test） | 非機能要件（応答時間・同時接続）の検証 | QAチーム | 100%（自動実行） |
| セキュリティテスト（Security Test） | 脆弱性・認証認可の検証 | セキュリティ担当 | 60%以上 |
| 受入テスト（UAT） | ビジネス要件の充足確認 | ステークホルダー | 手動 |

### 2.2 テストピラミッド

```
        /  UAT  \           ← 手動テスト（受入確認）
       /   E2E   \          ← Playwright（主要シナリオ）
      / 結合テスト  \        ← Supertest + TestContainers
     /  単体テスト   \       ← Jest + React Testing Library
    ━━━━━━━━━━━━━━━━━━
```

### 2.3 テスト種別詳細

#### 2.3.1 単体テスト

| 項目 | 内容 |
|------|------|
| 対象 | サービスクラス、ユーティリティ関数、Reactコンポーネント、カスタムフック |
| ツール | Jest、React Testing Library、ts-jest |
| カバレッジ目標 | 行カバレッジ 80%以上、ブランチカバレッジ 70%以上 |
| 実行タイミング | コミット時（pre-commit hook）、CI/CDパイプライン |

**フロントエンド単体テスト対象:**

| コンポーネント | テスト内容 |
|---------------|----------|
| RFCCreate.tsx | フォームバリデーション、必須項目チェック、送信処理 |
| RFCDetail.tsx | RFC詳細表示、ステータス遷移表示、承認ボタン制御 |
| CABDashboard.tsx | 審議一覧表示、フィルタリング、統計表示 |
| ChangeCalendar.tsx | カレンダー表示、イベント表示、衝突表示 |
| KPIDashboard.tsx | KPIグラフ描画、期間フィルタ、データ更新 |

**バックエンド単体テスト対象:**

| サービス | テスト内容 |
|---------|----------|
| ImpactAnalyzer | 依存関係解析、リスクレベル計算、影響ユーザー推定 |
| ConflictDetector | 重複変更検知、フリーズ期間検知、依存システム変更検知 |
| ApprovalWorkflow | 承認条件判定、承認数カウント、ステータス遷移 |
| NotificationService | テンプレート生成、送信先決定、通知種別判定 |

#### 2.3.2 結合テスト

| 項目 | 内容 |
|------|------|
| 対象 | API エンドポイント、データベース操作、キュー処理 |
| ツール | Jest、Supertest、TestContainers（PostgreSQL、Redis） |
| データベース | テスト用PostgreSQLコンテナ（テストごとにリセット） |
| 実行タイミング | プルリクエスト時、CI/CDパイプライン |

**結合テスト対象API:**

| エンドポイント | テスト内容 |
|--------------|----------|
| POST /api/v1/rfcs | RFC作成 → DB保存 → 衝突検知 → 通知キュー投入 |
| PATCH /api/v1/rfcs/:id/status | ステータス更新 → 承認ワークフロー → 通知送信 |
| POST /api/v1/cab/sessions | CABセッション作成 → 議題自動生成 → 召集通知 |
| POST /api/v1/cab/decisions | CAB承認/却下 → RFC状態更新 → 議事録生成 |
| GET /api/v1/calendar | 変更カレンダー取得 → フリーズ期間重畳 |
| GET /api/v1/reports/kpi | KPI集計 → グラフデータ生成 |

#### 2.3.3 E2Eテスト

| 項目 | 内容 |
|------|------|
| 対象 | ユーザーシナリオベースの全体フロー |
| ツール | Playwright |
| ブラウザ | Chromium、Firefox、WebKit |
| 実行タイミング | デプロイ前、リリース候補版 |

**E2Eテストシナリオ:**

| シナリオID | シナリオ名 | 概要 |
|-----------|----------|------|
| E2E-001 | 通常変更フルフロー | RFC起票 → CAB審議 → 承認 → 実施 → 完了 → PIR |
| E2E-002 | 緊急変更フロー | 緊急RFC起票 → ECAB承認 → 即時実施 → 事後レビュー |
| E2E-003 | 変更却下フロー | RFC起票 → CAB審議 → 却下 → 再申請 |
| E2E-004 | ロールバックフロー | RFC起票 → 承認 → 実施 → 失敗 → ロールバック |
| E2E-005 | フリーズ期間検知 | フリーズ期間設定 → 変更申請 → ブロック確認 |
| E2E-006 | 衝突検知フロー | 同一システム重複変更 → 衝突警告確認 |
| E2E-007 | KPIダッシュボード | 複数RFC処理 → KPI集計確認 → グラフ表示 |

#### 2.3.4 性能テスト

| 項目 | 内容 |
|------|------|
| 対象 | API応答時間、同時接続処理、データベース負荷 |
| ツール | k6、Artillery |
| 実行タイミング | リリース候補版、月次定期テスト |

詳細は「03_性能テスト計画(performance-test).md」を参照。

#### 2.3.5 セキュリティテスト

| テスト項目 | ツール | 対象 |
|-----------|-------|------|
| 依存パッケージ脆弱性 | npm audit、Snyk | package.json |
| SQLインジェクション | OWASP ZAP | 全APIエンドポイント |
| XSS（クロスサイトスクリプティング） | OWASP ZAP | フロントエンド入力フィールド |
| 認証・認可テスト | 手動 + 自動 | JWT検証、ロールベースアクセス制御 |
| CSRF対策 | 手動 | フォーム送信エンドポイント |
| ヘッダーセキュリティ | SecurityHeaders.com | HTTPレスポンスヘッダー |
| コンテナイメージ脆弱性 | Trivy | Docker イメージ |

## 3. テスト環境

### 3.1 環境構成

| 環境名 | 用途 | 構成 | データ |
|--------|------|------|--------|
| ローカル開発環境 | 単体テスト・結合テスト | Docker Compose（開発用） | テストデータ（シード） |
| CI環境 | 自動テスト実行 | GitHub Actions ランナー | テスト用コンテナDB |
| ステージング環境 | E2E・性能・セキュリティテスト | 本番同等Docker Compose | 匿名化済み本番相当データ |
| 本番環境 | スモークテスト | 本番構成 | 本番データ（読取のみ） |

### 3.2 テストデータ管理

| データ種別 | 管理方法 | 生成ツール |
|-----------|---------|-----------|
| マスターデータ | SQLシードファイル | backend/tests/seeds/ |
| テスト用ユーザー | フィクスチャファイル | backend/tests/fixtures/ |
| RFC テストデータ | ファクトリ関数 | backend/tests/factories/ |
| 大量データ | データ生成スクリプト | backend/tests/generators/ |

### 3.3 テスト用Docker Compose

```yaml
# docker-compose.test.yml
version: '3.9'
services:
  test-db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: cab_platform_test
      POSTGRES_USER: test_user
      POSTGRES_PASSWORD: test_password
    ports:
      - "5433:5432"
    tmpfs:
      - /var/lib/postgresql/data

  test-redis:
    image: redis:7-alpine
    ports:
      - "6380:6379"

  test-backend:
    build:
      context: ./backend
      dockerfile: Dockerfile.test
    environment:
      DATABASE_URL: postgresql://test_user:test_password@test-db:5432/cab_platform_test
      REDIS_URL: redis://test-redis:6379/0
      NODE_ENV: test
    depends_on:
      - test-db
      - test-redis
```

## 5. テスト実施基準

### 5.1 テスト開始基準

| テストレベル | 開始基準 |
|-------------|---------|
| 単体テスト | 対象モジュールの実装完了 |
| 結合テスト | 関連モジュールの単体テスト完了、テスト環境構築完了 |
| E2Eテスト | 全結合テスト合格、ステージング環境デプロイ完了 |
| 性能テスト | E2Eテスト合格、ステージング環境に本番相当データ投入完了 |
| セキュリティテスト | 結合テスト合格、ステージング環境デプロイ完了 |
| 受入テスト | 全テストレベル合格、リリース候補版準備完了 |

### 5.2 テスト完了基準

| テストレベル | 完了基準 |
|-------------|---------|
| 単体テスト | カバレッジ80%以上、全テストケース合格、重大バグ0件 |
| 結合テスト | 全APIエンドポイントテスト合格、データ整合性確認完了 |
| E2Eテスト | 全シナリオ合格、重大バグ0件、中程度バグ3件以下 |
| 性能テスト | 全目標値達成（応答2秒以内、同時30ユーザー処理可能） |
| セキュリティテスト | Critical/High脆弱性0件、Medium以下は対応計画策定済 |
| 受入テスト | ステークホルダー承認、全受入基準充足 |

### 5.3 バグ重要度分類

| 重要度 | 定義 | 対応期限 | 例 |
|--------|------|---------|-----|
| Critical | システム停止、データ損失 | 即時対応（4時間以内） | DB接続不能、認証バイパス |
| High | 主要機能障害、代替手段なし | 24時間以内 | RFC起票不可、承認処理失敗 |
| Medium | 機能一部障害、代替手段あり | 次スプリント | 表示崩れ、通知遅延 |
| Low | 軽微な不具合、利用に支障なし | バックログ | 文言誤り、レイアウト微調整 |

## 6. テストスケジュール

### 6.1 フェーズ別スケジュール

| フェーズ | 期間 | テスト内容 | 成果物 |
|---------|------|-----------|--------|
| Phase 1: 基盤テスト | Week 1-2 | 単体テスト（バックエンド） | テスト結果レポート、カバレッジレポート |
| Phase 2: UI テスト | Week 2-3 | 単体テスト（フロントエンド） | コンポーネントテスト結果 |
| Phase 3: 結合テスト | Week 3-4 | API結合テスト、DB結合テスト | 結合テスト結果レポート |
| Phase 4: E2Eテスト | Week 4-5 | 全シナリオE2Eテスト | E2Eテスト結果、スクリーンショット |
| Phase 5: 非機能テスト | Week 5-6 | 性能テスト、セキュリティテスト | 性能テストレポート、脆弱性レポート |
| Phase 6: 受入テスト | Week 6-7 | UATテスト | 受入テスト結果、承認書 |

### 6.2 CI/CDパイプライン統合

```
コミット → Lint/型チェック → 単体テスト → 結合テスト → ビルド → E2Eテスト → デプロイ
                                                                    ↓
                                                            性能テスト（週次）
                                                            セキュリティスキャン（週次）
```

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|-------|--------|------|
| テスト環境の構築遅延 | 高 | Docker Composeによる環境の即時構築、IaC管理 |
| テストデータ不足 | 中 | ファクトリ関数・シードデータの事前準備 |
| 外部サービス依存テストの不安定 | 中 | MSWによるAPIモック、Contract Testing |
| 性能テスト環境が本番と異なる | 中 | ステージング環境を本番同等構成で構築 |
| セキュリティ脆弱性の発見遅延 | 高 | CI/CDパイプラインへのセキュリティスキャン組込 |
