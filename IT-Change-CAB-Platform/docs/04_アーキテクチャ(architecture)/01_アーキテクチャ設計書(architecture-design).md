## 1. アーキテクチャ概要

### 1.1 アーキテクチャスタイル

本システムは**モノリシックレイヤードアーキテクチャ**を採用する。将来的なマイクロサービス化を見据えつつ、現在の開発チーム規模（少人数）と運用体制に最適なアーキテクチャとする。

| 判断ポイント | 選定理由 |
|------------|---------|
| モノリシック選定 | 開発チーム少人数（1-3名）、運用コスト最小化 |
| レイヤード選定 | 関心の分離、テスタビリティ確保 |
| 将来の分離点 | Service層をドメインごとに分離可能な設計 |

## 3. レイヤー設計

| 項目 | 内容 |

## 7. 可用性設計

### 7.1 障害対策

| 障害ポイント | 検知方法 | 対策 |
|------------|---------|------|
| バックエンドプロセス | PM2 watchdog | 自動再起動（cluster mode） |
| データベース | ヘルスチェック | WALアーカイブ + pg_basebackup |
| Redis | Sentinel | マスター/スレーブフェイルオーバー |
| ジョブワーカー | BullMQ stalledジョブ検知 | 自動リトライ + Dead Letter Queue |
| 外部API | タイムアウト + サーキットブレーカー | フォールバック処理 |

### 7.2 バックアップ戦略

| 対象 | 方式 | 頻度 | 保持期間 | 復旧手順 |
|------|------|------|---------|---------|
| PostgreSQL | pg_dump + WALアーカイブ | 日次 + 継続 | 3世代 + WAL 7日 | pg_restore / PITR |
| Redis | RDB + AOF | 日次 + 継続 | 3世代 | redis-cli restore |
| 添付ファイル | ファイルコピー | 日次 | 3世代 | ファイル復元 |
| 設定 | Git管理 | 変更時 | 永久 | git checkout |
