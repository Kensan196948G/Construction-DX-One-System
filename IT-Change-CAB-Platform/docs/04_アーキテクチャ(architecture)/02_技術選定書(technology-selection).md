## 2. フロントエンド技術選定

### 2.1 UIフレームワーク選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **React 18（選定）** | ★★★★★ | エコシステム充実、TypeScript親和性高、Ant Designとの組み合わせ実績豊富 |
| Vue 3 | ★★★★ | 学習コスト低いが、Ant Design for Vueはフォーム系でReact版より機能不足 |
| Angular | ★★★ | フルスタックだが学習コスト高、小規模チームではオーバースペック |
| Next.js | ★★★★ | SSR不要（社内SPA）、過剰な複雑性 |

### 2.2 UIコンポーネントライブラリ選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Ant Design 5（選定）** | ★★★★★ | 業務システム向けコンポーネント充実（Table/Form/DatePicker）、日本語対応良好 |
| MUI (Material UI) | ★★★★ | デザイン品質高いが、テーブル・フォーム系はAnt Designが優位 |
| Chakra UI | ★★★ | シンプルだが業務系コンポーネント不足 |
| shadcn/ui | ★★★ | カスタマイズ性高いが、組み込みコンポーネント数少 |

### 2.3 状態管理選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Zustand + React Query（選定）** | ★★★★★ | 軽量、TypeScript親和性高、サーバ状態とクライアント状態の分離が明確 |
| Redux Toolkit | ★★★★ | 実績あるが、ボイラープレートが多い |
| Recoil | ★★★ | Facebookだが安定性に課題 |
| Jotai | ★★★★ | 軽量だがReact Queryとの組み合わせ事例少 |

### 2.4 カレンダーライブラリ選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **FullCalendar 6（選定）** | ★★★★★ | 月/週/日ビュー完備、ドラッグ&ドロップ、iCal対応、React公式プラグイン |
| react-big-calendar | ★★★ | 基本機能のみ、カスタマイズ性低 |
| Ant Design Calendar | ★★★ | 月間ビューのみ、変更管理には機能不足 |

### 2.5 グラフライブラリ選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Recharts（選定）** | ★★★★★ | React ネイティブ、D3ベース、KPIダッシュボードに必要なグラフ種別を網羅 |
| Chart.js (react-chartjs-2) | ★★★★ | 軽量だがReact統合がラッパー経由 |
| Nivo | ★★★★ | 美しいがカスタマイズ性でRechartsに劣る |
| ECharts (echarts-for-react) | ★★★★ | 高機能だが学習コスト高 |

## 3. バックエンド技術選定

### 3.1 ランタイム選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Node.js 20 LTS（選定）** | ★★★★★ | フロントエンドとの言語統一（TypeScript）、非同期I/O性能、npm エコシステム |
| Deno | ★★★ | セキュリティ良いがエコシステム未成熟 |
| Python (FastAPI) | ★★★★ | 高性能だが言語統一のメリットを失う |
| Go (Gin) | ★★★★ | 高性能だが開発速度でNode.jsに劣る（少人数チーム） |

### 3.2 APIフレームワーク選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Express 4（選定）** | ★★★★★ | 最大のエコシステム、ミドルウェア充実、学習コスト最低 |
| Fastify | ★★★★ | 高性能だがExpress互換性に課題 |
| NestJS | ★★★★ | 構造化されているが小規模には過剰 |
| Koa | ★★★ | 軽量だがミドルウェアエコシステムがExpress未満 |

### 3.3 認証ライブラリ選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Passport.js + jsonwebtoken（選定）** | ★★★★★ | 認証戦略の柔軟性（Local/JWT/SAML/OIDC）、将来のSSO対応 |
| Auth0 SDK | ★★★★ | 簡単だが外部依存、コスト発生 |
| NextAuth | ★★★ | Next.js専用 |

### 3.4 データベースアクセス選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Knex.js（選定）** | ★★★★★ | SQL直接制御可能、マイグレーション内蔵、学習コスト低、十分な型安全性 |
| Prisma | ★★★★ | 型安全性最高だが、複雑なクエリでの制約あり |
| TypeORM | ★★★ | Active Recordパターンだが不安定性の報告あり |
| Drizzle | ★★★★ | 新しく高性能だが実績不足 |
| Sequelize | ★★★ | 古く、TypeScript対応が不十分 |

### 3.5 バリデーション選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Zod（選定）** | ★★★★★ | TypeScript first、フロント・バックエンド共用可能、推論型が優秀 |
| Joi | ★★★★ | 実績あるがTypeScript推論が弱い |
| Yup | ★★★ | フォームバリデーション向け |
| express-validator | ★★★ | Express専用、スキーマ共有不可 |

## 4. データベース技術選定

### 4.1 RDBMS選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **PostgreSQL 16（選定）** | ★★★★★ | JSON型サポート（投票記録等）、配列型、全文検索、拡張性、OSS |
| MySQL 8 | ★★★★ | 実績あるがJSON/配列操作でPostgreSQLに劣る |
| SQLite | ★★ | 組み込み用途、並行処理に制約 |
| MariaDB | ★★★ | MySQL互換だがPostgreSQLの方がJSON処理に優位 |

### 4.2 PostgreSQL選定の決め手

| 機能 | 活用箇所 |
|------|---------|
| JSONB型 | CAB投票記録、通知テンプレート、設定値 |
| 配列型 | 対象システム一覧（VARCHAR[]）、ブロック対象変更種別 |
| 全文検索 | RFC検索（タイトル、説明文のFull Text Search） |
| CHECK制約 | 状態遷移の整合性保証 |
| UUID生成 | gen_random_uuid()によるID生成 |
| TIMESTAMPTZ | タイムゾーン対応の日時管理 |
| トリガー | updated_at自動更新、監査ログ自動記録 |

### 4.3 キャッシュ/キュー選定

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Redis 7（選定）** | ★★★★★ | セッション管理、キャッシュ、BullMQバックエンドの3役兼務 |
| Memcached | ★★★ | キャッシュのみ、キュー非対応 |

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **BullMQ（選定）** | ★★★★★ | Redis上で動作、TypeScript native、リトライ/優先度/スケジュール対応 |
| Agenda | ★★★ | MongoDB依存 |
| node-cron | ★★ | インメモリ、永続性なし |
| RabbitMQ | ★★★★ | 高機能だが運用コスト高、本規模では過剰 |

## 5. インフラ技術選定

### 5.1 コンテナ化

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Docker + Docker Compose（選定）** | ★★★★★ | 環境再現性、デプロイ簡便、開発・本番環境統一 |
| Kubernetes | ★★★ | 小規模に対しオーバースペック |
| Podman | ★★★★ | Docker互換だが普及度で劣る |

### 5.2 Webサーバ/プロキシ

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **Nginx（選定）** | ★★★★★ | SSL終端、静的配信、リバースプロキシ、レート制限の実績 |
| Caddy | ★★★★ | 自動HTTPS、設定簡単だがカスタマイズ性でNginx未満 |
| Apache | ★★★ | 高機能だが設定複雑 |

### 5.3 プロセス管理

| 候補 | 評価 | 選定理由 |
|------|------|---------|
| **PM2（選定）** | ★★★★★ | クラスターモード、ログ管理、ウォッチドッグ、ゼロダウンタイムリロード |
| systemd | ★★★ | Linux標準だがNode.js固有の機能不足 |
| forever | ★★ | 非推奨 |

## 7. 技術的制約

### 7.1 環境制約

| 制約 | 詳細 | 影響 |
|------|------|------|
| オンプレミスホスティング | クラウドサービス（AWS/Azure PaaS）は使用しない | Docker Compose構成に限定 |
| 社内ネットワーク | インターネット経由のSaaS利用制限あり | 外部API連携にプロキシが必要な場合あり |
| SSL証明書 | 社内CA発行の証明書を使用 | ブラウザ信頼設定が必要 |

### 7.2 運用制約

| 制約 | 詳細 | 対応策 |
|------|------|--------|
| 運用チーム1-2名 | 大規模インフラ運用は不可 | Docker Composeによる簡易運用 |
| メンテナンスウィンドウ | 業務時間外のみ | ゼロダウンタイムデプロイ（PM2 reload） |
| バックアップ | 社内NASへの保存 | pg_dump + rsync自動化 |

### 7.3 技術的負債リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| Node.js EOL | 中 | LTS版使用、2年ごとのアップグレード計画 |
| 依存パッケージ脆弱性 | 高 | npm audit週次実行、Dependabot |
| Ant Design破壊的変更 | 中 | メジャーバージョン固定、十分なテスト |
| PostgreSQL EOL | 低 | LTS相当（5年サポート） |
