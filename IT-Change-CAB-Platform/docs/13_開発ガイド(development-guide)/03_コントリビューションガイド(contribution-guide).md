## 1. コントリビューションの流れ

### 1.1 全体フロー

```
1. GitHub Issue 確認・作成
   │
2. ブランチ作成
   │
3. 実装・テスト
   │
4. プルリクエスト作成
   │
5. コードレビュー
   │
6. CI パス確認
   │
7. マージ
```

---

## 2. Issue管理

### 2.1 Issueの種類

| ラベル | 用途 | テンプレート |
|-------|------|------------|
| `feature` | 新機能要望 | Feature Request |
| `bug` | バグ報告 | Bug Report |
| `enhancement` | 既存機能改善 | Enhancement |
| `documentation` | ドキュメント修正 | - |
| `security` | セキュリティ関連 | Security Issue |

### 2.2 Issue起票ルール

| ルール | 説明 |
|-------|------|
| 事前検索 | 同様のIssueが既にないか検索する |
| テンプレート使用 | Issue テンプレートを使用して起票する |
| 再現手順 | バグの場合は再現手順を明記する |
| 優先度ラベル | `priority:high` / `priority:medium` / `priority:low` を付与 |
| アサイン | 作業者が決まったらアサインする |

## 3. ブランチ戦略

### 3.1 ブランチ命名規則

| ブランチ種別 | 命名規則 | 例 |
|-------------|---------|-----|
| 機能追加 | `feature/<issue番号>-<概要>` | `feature/42-rfc-form-validation` |
| バグ修正 | `fix/<issue番号>-<概要>` | `fix/55-cab-approval-count` |
| 緊急修正 | `hotfix/<概要>` | `hotfix/security-patch-jwt` |
| ドキュメント | `docs/<概要>` | `docs/update-api-reference` |
| リファクタリング | `refactor/<概要>` | `refactor/impact-analyzer` |

### 3.2 ブランチ作成手順

```bash
# 1. mainブランチを最新に更新
git checkout main
git pull origin main

# 2. 作業ブランチを作成
git checkout -b feature/42-rfc-form-validation

# 3. 作業開始
```

### 3.3 ブランチのライフサイクル

| フェーズ | アクション |
|---------|-----------|
| 作成 | mainから分岐 |
| 開発 | コミット・プッシュ |
| PR作成 | レビュー依頼 |
| マージ | mainにマージ |
| 削除 | マージ後にブランチ削除 |

## 4. コミット規約

### 4.1 Conventional Commits 形式

```
<type>(<scope>): <subject>

[body]

[footer]
```

### 4.2 コミットメッセージのルール

| ルール | 説明 |
|-------|------|
| subject | 50文字以内、命令形（日本語の場合は体言止め） |
| body | 変更の理由・背景を記載（任意） |
| footer | 関連Issue番号を記載（`Closes #42`） |
| 1コミット1変更 | 1つの論理的変更につき1コミット |

### 4.3 コミットメッセージ例

```
feat(rfc): RFC起票フォームにバリデーション追加

- タイトルの必須チェックを追加
- 実施日の未来日チェックを追加
- ロールバック計画の必須チェック（通常/重大変更時）

Closes #42
```

## 5. プルリクエスト（PR）

### 5.1 PR作成ルール

| ルール | 説明 |
|-------|------|
| PRサイズ | 変更行数は300行以内を目安（大きい場合は分割） |
| 1PR1機能 | 1つのPRに複数の無関係な変更を含めない |
| テスト必須 | 新機能・バグ修正には対応するテストを含める |
| ドキュメント | API変更時はAPI仕様書も更新する |
| スクリーンショット | UI変更時はスクリーンショットを添付する |

### 5.2 PRテンプレート

```markdown
## 概要
<!-- 変更内容の概要を記載 -->

## 関連Issue
Closes #XX

## 変更種別
- [ ] 新機能（feature）
- [ ] バグ修正（fix）
- [ ] リファクタリング（refactor）
- [ ] ドキュメント（docs）
- [ ] テスト（test）
- [ ] その他

## 変更内容
<!-- 具体的な変更内容を箇条書きで記載 -->

## テスト
<!-- 追加・変更したテストの概要 -->
- [ ] 単体テスト追加/更新
- [ ] 結合テスト追加/更新
- [ ] 手動テスト実施

## スクリーンショット（UI変更の場合）
<!-- Before/After のスクリーンショット -->

## チェックリスト
- [ ] コーディング規約に準拠している
- [ ] テストが全て通過している
- [ ] 型チェックが通過している
- [ ] ESLintエラーがない
- [ ] 必要なドキュメントを更新している
```

### 5.3 PR作成手順

```bash
# 1. 変更をコミット・プッシュ
git add .
git commit -m "feat(rfc): RFC起票フォームにバリデーション追加"
git push origin feature/42-rfc-form-validation

# 2. GitHub上でPR作成
# または gh コマンドで作成
gh pr create --title "feat(rfc): RFC起票フォームにバリデーション追加" \
  --body "Closes #42"
```

## 6. コードレビュー

### 6.1 レビュー基準

| 観点 | 確認ポイント |
|------|------------|
| 機能性 | 要件を満たしているか、エッジケースは考慮されているか |
| 可読性 | コードが理解しやすいか、命名は適切か |
| 保守性 | 変更しやすい構造か、重複はないか |
| テスト | テストは十分か、テストケースは適切か |
| セキュリティ | 入力検証、認証認可、SQLインジェクション対策 |
| 性能 | N+1クエリ、不要な再レンダリング、メモリリーク |
| 型安全性 | TypeScriptの型が適切に使用されているか |
| エラーハンドリング | エラーが適切に処理されているか |

### 6.2 レビューの進め方

| ステップ | 内容 |
|---------|------|
| 1. 概要理解 | PR説明と関連Issueを読み、変更の目的を理解する |
| 2. 差分確認 | 変更ファイルを順に確認する |
| 3. テスト確認 | テストが適切に追加されているか確認する |
| 4. ローカル確認 | 必要に応じてローカルで動作確認する |
| 5. フィードバック | コメントを残す（指摘・質問・改善提案） |
| 6. 承認/修正依頼 | 問題なければApprove、修正必要ならRequest Changes |

### 6.3 レビューコメントの書き方

| プレフィックス | 意味 | 例 |
|-------------|------|-----|
| `[must]` | 必須修正 | `[must] SQLインジェクションの可能性があります` |
| `[should]` | 強く推奨 | `[should] エラーハンドリングを追加してください` |
| `[nit]` | 軽微な指摘 | `[nit] 変数名を `rfcList` にすると意図が明確です` |
| `[question]` | 質問 | `[question] この条件分岐の意図は何ですか？` |
| `[praise]` | 良い点 | `[praise] テストケースが充実していて良いです` |

### 6.4 レビュー応答のルール

| ルール | 説明 |
|-------|------|
| 24時間以内 | レビュー依頼から24時間以内にレビューを開始する |
| 修正対応 | `[must]` コメントは全て対応する |
| 議論 | 指摘に異論がある場合はコメントで議論する |
| 再レビュー | 修正後に再レビューを依頼する |

## 7. CIパイプライン

### 7.1 PR時のCIチェック

| チェック | 内容 | 必須 |
|---------|------|------|
| lint | ESLint チェック | 必須 |
| type-check | TypeScript 型チェック | 必須 |
| unit-test | 単体テスト実行 | 必須 |
| integration-test | 結合テスト実行 | 必須 |
| build | ビルド成功確認 | 必須 |
| coverage | カバレッジ閾値チェック（80%以上） | 必須 |
| security | npm audit | 警告 |

### 7.2 マージ条件

| 条件 | 説明 |
|------|------|
| CIチェック全パス | 全てのCIチェックが成功していること |
| レビュー承認 | 1名以上のApproveがあること |
| コンフリクトなし | mainブランチとのコンフリクトがないこと |
| 会話解決 | 全てのレビューコメントが解決済みであること |

## 8. マージとリリース

### 8.1 マージ方法

| 方法 | 使用場面 |
|------|---------|
| Squash and Merge | 通常の機能追加・バグ修正（推奨） |
| Merge Commit | 複数の意味のあるコミットを保持したい場合 |
| Rebase and Merge | 使用しない |

### 8.2 マージ後の作業

1. ブランチの削除（GitHub上で自動削除設定推奨）
2. 関連Issueの自動クローズ確認
3. ステージング環境への自動デプロイ確認

## 9. セキュリティに関する注意事項

| 項目 | ルール |
|------|-------|
| 機密情報 | パスワード・APIキー・トークンをコードに含めない |
| .env ファイル | `.gitignore`に含め、リポジトリにコミットしない |
| 依存パッケージ | `npm audit` で脆弱性を定期確認する |
| 入力値検証 | 全てのAPIエンドポイントで入力値を検証する |
| SQLクエリ | パラメータ化クエリを使用する（文字列結合禁止） |
