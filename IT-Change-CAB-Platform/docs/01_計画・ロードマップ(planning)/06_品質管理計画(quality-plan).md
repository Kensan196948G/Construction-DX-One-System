## 1. 品質管理方針

### 1.1 目的

ISO20000変更管理プロセスに準拠したプラットフォームとして、機能品質・非機能品質の両面で高い水準を維持し、運用開始後も継続的に品質を向上させる。

### 1.2 品質方針

1. **要件準拠性** - 要件定義書（REQ-CAB-001）の全機能要件を満たす
2. **信頼性** - システム稼働率99.5%以上を維持する
3. **セキュリティ** - OWASP Top 10に対する防御を実装する
4. **保守性** - コードの可読性・テスタビリティを確保する
5. **ユーザビリティ** - IT部門以外の一般ユーザでも直感的に操作できる

### 1.3 品質管理体制

| 役割 | 担当 | 責務 |
|------|------|------|
| 品質管理責任者 | PM | 品質計画の策定・監視・報告 |
| QAリード | QA（IT部門員C） | テスト計画・実行・品質指標の測定 |
| コードレビュア | LE + 開発チーム | コードレビュー、設計レビュー |
| セキュリティレビュア | IT部門 セキュリティ担当 | セキュリティテスト、脆弱性評価 |

## 3. 品質基準

### 3.1 コーディング品質基準

| 基準項目 | 基準内容 | 適用範囲 | 検証方法 |
|---------|---------|---------|---------|
| TypeScript Strict Mode | `strict: true`を全プロジェクトで有効化 | 全TypeScriptコード | tsconfig.json, CI |
| ESLint ルール | `@typescript-eslint/recommended` + カスタムルール適用 | 全TypeScriptコード | CI（eslint） |
| Prettier フォーマット | 統一フォーマット（セミコロン、シングルクォート、2スペース） | 全コード | CI（prettier --check） |
| 命名規則 | camelCase（変数/関数）、PascalCase（型/コンポーネント）、UPPER_SNAKE_CASE（定数） | 全コード | コードレビュー |
| 関数行数制限 | 50行以内（超過時は分割を検討） | 全コード | コードレビュー |
| ファイル行数制限 | 300行以内（超過時は分割を検討） | 全コード | コードレビュー |
| コメント | 公開APIにはJSDocコメント必須 | バックエンドAPI, ライブラリ | コードレビュー |
| テスト必須 | 新規関数/クラスにはユニットテスト必須 | 全ビジネスロジック | コードレビュー |

### 3.2 API品質基準

| 基準項目 | 基準内容 | 検証方法 |
|---------|---------|---------|
| RESTful設計 | リソース指向URL、適切なHTTPメソッド・ステータスコード | API設計レビュー |
| 入力バリデーション | 全APIエンドポイントでZodスキーマによるバリデーション | 単体テスト |
| エラーレスポンス | 統一フォーマット `{error: {code, message, details}}` | 単体テスト |
| 認証・認可 | 全エンドポイント（ヘルスチェック除く）で認証必須 | 結合テスト |
| レスポンス時間 | P95 2秒以内 | 性能テスト |
| ページネーション | 一覧APIは全てページネーション対応 | 単体テスト |
| OpenAPI仕様 | 全エンドポイントの仕様を文書化 | API仕様レビュー |

### 3.3 フロントエンド品質基準

| 基準項目 | 基準内容 | 検証方法 |
|---------|---------|---------|
| アクセシビリティ | Lighthouse Accessibilityスコア80以上 | Lighthouse |
| レスポンシブ対応 | 最小幅1024px以上で正常表示 | 目視確認 |
| エラーハンドリング | API通信エラー時のユーザフレンドリーなメッセージ表示 | 結合テスト |
| ローディング表示 | 非同期処理中のスピナー/スケルトン表示 | 目視確認 |
| フォームバリデーション | クライアントサイドバリデーション（リアルタイム） | 単体テスト |
| バンドルサイズ | 初期ロード1MB以内（gzip後） | Vite build |

### 3.4 セキュリティ品質基準

| 基準項目 | 基準内容 | 検証方法 |
|---------|---------|---------|
| OWASP Top 10対策 | 全10項目に対する防御実装 | セキュリティテスト |
| 認証 | JWT + Refresh Token、トークン有効期限15分 | セキュリティレビュー |
| パスワード | bcrypt（コスト12以上）によるハッシュ化 | コードレビュー |
| SQL Injection防御 | ORM使用、パラメタライズドクエリ | コードレビュー |
| XSS防御 | React DOM自動エスケープ、CSPヘッダ設定 | セキュリティテスト |
| CSRF防御 | CSRFトークン（SameSite=Strict） | セキュリティテスト |
| 機密情報管理 | 環境変数経由、ハードコード禁止、ログマスキング | コードレビュー |
| HTTPS | 全通信TLS 1.2以上 | インフラ設定レビュー |
| 依存パッケージ | npm audit high/critical 0件 | CI（npm audit） |
| CORS | 許可オリジンの明示的指定 | 設定レビュー |

### 3.5 データベース品質基準

| 基準項目 | 基準内容 | 検証方法 |
|---------|---------|---------|

## 7. 不具合管理

### 7.1 バグ重要度分類

| 重要度 | 定義 | 対応期限 | 例 |
|--------|------|---------|-----|
| Critical | サービス停止、データ損失、セキュリティ脆弱性 | 24時間以内 | 認証バイパス、DB破損 |
| High | 主要機能が動作しない、回避策なし | 3営業日以内 | RFC起票不可、CAB承認不可 |
| Medium | 一部機能の不具合、回避策あり | 次スプリント | 検索フィルタの不具合 |
| Low | 軽微なUI問題、改善提案 | バックログ | 表示崩れ、文言修正 |

### 7.2 バグライフサイクル

```
新規 → 確認済み → 対応中 → レビュー中 → 検証中 → 完了
  │                                          │
  └→ 仕様通り / 再現不可 → クローズ          └→ 再オープン → 対応中
```

| 項目 | 内容 |
|------|------|
| タイトル | 簡潔な不具合の説明 |

## 9. CI/CDによる品質自動化

### 9.1 CI パイプライン品質チェック

```
PR作成/更新
  │
  ├─ 1. TypeScript コンパイル（tsc --noEmit）
  │    └─ 型エラー → PR ブロック
  │
  ├─ 2. ESLint 静的解析
  │    └─ エラー → PR ブロック
  │
  ├─ 3. Prettier フォーマットチェック
  │    └─ 差分あり → PR ブロック
  │
  ├─ 4. 単体テスト実行（Jest / Vitest）
  │    └─ 失敗 → PR ブロック
  │
  ├─ 5. カバレッジチェック
  │    └─ 閾値未満 → PR ブロック
  │
  ├─ 6. npm audit（セキュリティ）
  │    └─ High/Critical → 警告（ブロックはオプション）
  │
  └─ 7. ビルド確認
       └─ 失敗 → PR ブロック

全チェック合格 → レビュー可能
```

### 9.2 CDパイプライン品質ゲート

```
main ブランチマージ
  │
  ├─ 1. 全CI チェック再実行
  ├─ 2. E2E テスト実行（Playwright）
```
