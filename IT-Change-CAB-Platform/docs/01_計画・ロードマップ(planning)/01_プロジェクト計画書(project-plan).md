## 1. プロジェクト概要

### 1.1 プロジェクト名

IT変更管理・リリース自動化プラットフォーム（IT-Change-CAB-Platform）

### 1.2 背景

みらい建設工業のIT環境は、オンプレミス（Active Directory、ファイルサーバ、業務システム）とクラウド（Azure、Microsoft 365、HENGEONE）が混在するハイブリッド構成であり、各システム間の依存関係が複雑化している。現状の変更管理では以下の重大な課題が発生している。

| 課題領域 | 現状の問題 | 業務影響 |
|----------|-----------|---------|
| 変更申請プロセス | 口頭・メール・紙による非定型申請 | 変更内容の追跡不能、証跡不足 |
| CAB承認プロセス | 承認者・審議内容の記録が不在 | ISO20000監査指摘リスク |
| 影響分析 | 担当者の属人的判断に依存 | システム障害の見落とし |
| ロールバック計画 | 事前準備が不十分なケースが頻発 | 障害復旧の長期化 |
| 変更衝突管理 | 複数変更の競合を手動で確認 | 予期しないサービス停止 |
| コンプライアンス | ISO20000準拠の証跡が不完全 | 認証維持リスク |

### 1.3 目的

RFC（Request for Change）の起票からCAB承認・変更実施・事後レビュー（PIR）までの全変更プロセスを自動化・標準化し、以下の目標を達成する。

1. **ISO20000変更管理プロセスへの完全準拠** - 全変更に対する監査証跡の自動生成
2. **変更リスクの可視化と低減** - 影響分析エンジンによる依存関係・リスクの自動評価
3. **変更承認リードタイムの短縮** - CABオンライン承認により承認待ち時間を50%削減
4. **変更成功率の向上** - 目標変更成功率95%以上（現状推定80%）
5. **運用効率の改善** - 変更管理工数を60%削減

## 2. スコープ

### 2.1 スコープ内

| カテゴリ | 内容 |
|----------|------|
| **RFC管理** | RFC起票フォーム、変更種別自動分類、影響分析テンプレート、ロールバック計画必須化 |
| **CAB管理** | CABスケジュール管理、審議議題自動生成、オンライン承認、ECAB対応、議事録自動生成 |
| **変更実施管理** | 実施チェックリスト、実施記録、フリーズ期間管理、ロールバック実行管理 |
| **事後レビュー** | PIR記録、変更成功率KPIダッシュボード、変更カレンダー、トレンド分析 |
| **影響分析** | システム依存関係マップに基づく自動影響分析、変更衝突検知 |
| **通知連携** | Microsoft Teams Webhook通知、メール通知、承認リンク |
| **対象システム** | AD、Entra ID、Exchange Online、M365、HENGEONE、ファイルサーバ、DeskNet'sNeo、AppSuite、ネットワーク機器、Azureリソース |

### 2.2 スコープ外

| 項目 | 理由 |
|------|------|
| CMDB構築 | 既存ITSM-Systemで管理（APIで連携） |
| インシデント管理 | 別プロジェクト（ITSM-System）で実装済み |
| 資産管理 | IT-Asset-Managementプロジェクトで実装 |
| 変更の自動実行（IaC） | Phase 4で検討（今期スコープ外） |

### 2.3 前提条件

| No. | 前提条件 | 備考 |
|-----|---------|------|
| 1 | IT部門メンバー（最大30名）が本システムを利用する | 各部門代表も一部利用 |
| 2 | 本番環境はDocker Compose構成でオンプレミスサーバに展開 | 将来的にAzure移行も検討 |
| 3 | ITSM-SystemのAPIが利用可能 | インシデント・CI情報の連携 |
| 4 | Microsoft Teams WebhookおよびExchange Online APIが利用可能 | 通知連携 |
| 5 | PostgreSQL 16、Redis 7、Node.js 20 LTSが利用可能 | インフラ要件 |

### 2.4 制約条件

| No. | 制約条件 | 影響 |
|-----|---------|------|
| 1 | 開発リソースは実質1〜2名（IT部門兼務） | 開発期間に影響 |
| 2 | 社外SaaSへの機密データ保存不可 | オンプレミス展開を優先 |
| 3 | 既存認証基盤（AD/Entra ID/HENGEONE）との統合が必須 | SSO連携の実装が必要 |
| 4 | ISO20000次回監査（2026年12月）までに基本機能を稼働させる | Phase 2完了が必須 |

## 7. リスク管理方針

### 7.1 リスク管理プロセス

```
識別 → 分析（影響度×発生確率） → 対応策策定 → 監視 → レビュー（月次）
```

### 7.2 主要リスクサマリ

| リスクID | リスク | 影響度 | 発生確率 | 対応方針 |
|----------|--------|:---:|:---:|----------|
| R-001 | 開発リソース不足（兼務による遅延） | 高 | 高 | フェーズ分割で段階リリース、MVP優先 |
| R-002 | 既存システム連携の技術的困難 | 中 | 中 | 早期POCで検証、フォールバック設計 |
| R-003 | ISO20000監査に間に合わない | 高 | 低 | Phase 2完了をデッドラインとし優先管理 |
| R-004 | ユーザ（各部門）の利用定着が進まない | 中 | 中 | 段階的導入、トレーニング計画 |
| R-005 | セキュリティ脆弱性 | 高 | 低 | セキュリティレビュー必須、OWASP準拠 |

詳細は「04_リスク管理計画(risk-management).md」を参照。
