## 1. リスク管理方針

### 1.1 目的

プロジェクト遂行において発生しうるリスクを事前に識別・評価し、適切な対応策を講じることで、プロジェクト目標の達成を確保する。

### 1.2 リスク管理プロセス

```
1. 識別 → 2. 分析（定性・定量） → 3. 対応策策定 → 4. 監視・コントロール → 5. レビュー
     ↑                                                                    │
     └────────────────────────────────────────────────────────────────────┘
                              継続的サイクル
```

### 1.3 リスク管理体制

| 役割 | 担当 | 責務 |
|------|------|------|
| リスクオーナー | PO（IT部門長） | リスク受容判断、エスカレーション対応 |
| リスク管理者 | PM | リスク識別・評価・対応策実行管理 |
| リスク報告者 | 全メンバー | リスク発見時の報告 |

### 1.4 リスクレビューサイクル

| 活動 | 頻度 | 参加者 | 内容 |
|------|------|--------|------|
| リスクレビュー | 隔週（進捗会議内） | PM, 開発チーム | リスク状態の確認・更新 |
| リスク月次報告 | 月次 | PO, PM | 月次リスク評価レポート |
| 臨時リスク評価 | 随時 | PM, 関連者 | 新規リスク発生時 |

## 2. リスク評価基準

### 2.1 影響度基準

| レベル | 影響度 | スケジュール影響 | コスト影響 | 品質影響 |
|:---:|--------|----------------|-----------|---------|
| 5 | 致命的 | 2ヶ月以上遅延 | 予算50%超過 | サービス停止・データ損失 |
| 4 | 重大 | 1〜2ヶ月遅延 | 予算30%超過 | 重大機能欠損 |
| 3 | 中程度 | 2〜4週間遅延 | 予算15%超過 | 一部機能欠損 |
| 2 | 軽微 | 1〜2週間遅延 | 予算5%超過 | 軽微な品質低下 |
| 1 | 無視可能 | 1週間未満遅延 | 予算内 | 影響なし |

### 2.2 発生確率基準

| レベル | 発生確率 | 確率範囲 | 説明 |
|:---:|--------|---------|------|
| 5 | ほぼ確実 | 80%以上 | 過去に頻繁に発生 |
| 4 | 高い | 60〜79% | 発生する可能性が高い |
| 3 | 中程度 | 40〜59% | 発生する可能性がある |
| 2 | 低い | 20〜39% | 発生する可能性は低い |
| 1 | まれ | 20%未満 | ほぼ発生しない |

### 2.3 リスク評価マトリクス

リスクスコア = 影響度 x 発生確率

|  | 影響度1 | 影響度2 | 影響度3 | 影響度4 | 影響度5 |
|---|:---:|:---:|:---:|:---:|:---:|
| **確率5** | 5 | 10 | **15** | **20** | **25** |
| **確率4** | 4 | 8 | **12** | **16** | **20** |
| **確率3** | 3 | 6 | 9 | **12** | **15** |
| **確率2** | 2 | 4 | 6 | 8 | 10 |
| **確率1** | 1 | 2 | 3 | 4 | 5 |

| スコア範囲 | リスクレベル | 対応方針 |
|-----------|------------|---------|
| 15〜25 | 極高（赤） | 即時対応必須。PO判断、エスカレーション |
| 10〜14 | 高（橙） | 優先対応。PM主導で対策実行 |
| 5〜9 | 中（黄） | 対応計画策定。監視強化 |
| 1〜4 | 低（緑） | 受容可能。定期監視 |

## 4. リスク対応策

### 4.1 極高リスクの対応策

#### O-001: 開発リソース不足（兼務）

| 項目 | 内容 |
|------|------|
| リスクスコア | 16（極高） |
| 対応戦略 | 軽減 |
| 対応策1 | フェーズ分割によるMVP優先アプローチの採用。Phase 2（コア機能）を最優先とし、Phase 3以降は状況に応じて調整 |
| 対応策2 | 開発メンバーの通常IT運用業務を一時的に軽減するようPOが部門内で調整 |
| 対応策3 | AIコーディングアシスタント（Claude Code等）の活用により生産性を向上 |
| 対応策4 | 必要に応じて外部パートナーへの一部タスク委託を検討 |
| トリガー | 2週間連続でスプリント目標の70%未満の達成率 |
| 監視方法 | スプリントベロシティの追跡（週次） |
| 対応者 | PM, PO |

### 4.2 高リスクの対応策

#### O-004: 変更管理プロセスへの抵抗

| 項目 | 内容 |
|------|------|
| リスクスコア | 12（高） |
| 対応戦略 | 軽減 |
| 対応策1 | 段階的導入（まず通常変更のみ → 全変更種別へ拡大） |
| 対応策2 | トレーニングとハンズオンワークショップの実施 |
| 対応策3 | アーリーアダプター（協力的な部門）からの先行導入 |
| 対応策4 | 経営層からのトップダウンでの利用推進メッセージ |
| トリガー | 本番稼働後1ヶ月でRFC起票率が50%未満 |
| 監視方法 | RFC起票件数・ユーザログイン率の追跡（週次） |
| 対応者 | PM, PO |

#### S-001: Phase 2遅延によるISO監査対応不可

| 項目 | 内容 |
|------|------|
| リスクスコア | 10（高） |
| 対応戦略 | 回避 |
| 対応策1 | Phase 2のクリティカルパスタスクを最優先で管理 |
| 対応策2 | Phase 2デッドライン（2026-07-26）に4週間のバッファを設定。最悪ケースでも2026-08-23までにMVPリリース |
| 対応策3 | ISO20000監査で最低限必要な機能を特定し、最小セットを先行実装 |
| 対応策4 | 監査コンサルタントとの事前調整により、プラットフォーム導入途中でも証跡を示す方法を確認 |
| トリガー | Phase 2の進捗がW4時点で50%未満 |
| 監視方法 | バーンダウンチャート（週次） |
| 対応者 | PM |

#### S-002: 要件追加によるスコープクリープ

| 項目 | 内容 |
|------|------|
| リスクスコア | 12（高） |
| 対応戦略 | 軽減 |
| 対応策1 | 変更要求プロセスの厳格運用。全要件変更はPO承認必須 |
| 対応策2 | 追加要件は次フェーズ（Phase 5以降）のバックログに積む |
| 対応策3 | MoSCoW法によるスコープ優先度の明確化 |
| トリガー | 新規要件が3件以上追加された場合 |
| 監視方法 | 変更要求ログの確認（隔週） |
| 対応者 | PM, PO |

#### S-003: テスト工程の圧縮

| 項目 | 内容 |
|------|------|
| リスクスコア | 12（高） |
| 対応戦略 | 軽減 |
| 対応策1 | TDD（テスト駆動開発）の採用により実装とテストを並行実施 |
| 対応策2 | CI/CDパイプラインによる自動テストで手動テスト工数を削減 |
| 対応策3 | テスト工程は最低限2週間を確保。実装遅延時は機能を削減 |
| トリガー | 実装完了がテスト開始予定日を超過 |
| 監視方法 | 実装タスクの完了状況（週次） |
| 対応者 | PM, QA |

## 6. コンティンジェンシープラン

### 6.1 最悪シナリオと対応

| シナリオ | 発生条件 | コンティンジェンシー |
|---------|---------|-------------------|
| Phase 2がISO監査に間に合わない | 2026年8月末までにMVP未リリース | 紙ベース+Excelでの変更管理証跡を並行運用し、監査に提出。システムは次回監査までに完成 |
| リードエンジニアの離脱 | 離職・長期休暇 | ナレッジドキュメントによる引継ぎ、外部パートナーの緊急調達（2週間以内） |
| 本番環境のサーバ障害 | ハードウェア故障 | Docker Compose構成を別サーバに展開（RTO 4時間以内）。日次バックアップからDB復元 |
| セキュリティインシデント（認証突破） | 認証脆弱性の悪用 | システム緊急停止、インシデント対応チーム召集、脆弱性パッチ適用後に復旧 |
