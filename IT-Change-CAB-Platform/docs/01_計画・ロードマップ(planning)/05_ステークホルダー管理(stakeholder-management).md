## 1. 目的

プロジェクトに関与する全ステークホルダーを識別し、各ステークホルダーの期待・影響力・関心度を分析したうえで、適切なコミュニケーション戦略を策定・実行する。

## 4. ステークホルダー詳細分析

### 4.1 SH-01: 経営層（プロジェクトスポンサー）

| 項目 | 内容 |
|------|------|
| 影響力 | 極めて高い（予算承認、プロジェクト存続判断） |
| 関心度 | 高い（コンプライアンス・投資対効果） |
| 現在の態度 | 支持的（ISO20000対応の必要性を認識） |
| 期待 | ISO20000監査の通過、IT統制の可視化、コスト効率 |
| 懸念 | 追加投資の必要性、プロジェクト遅延リスク |
| 管理戦略 | 月次経営報告、マイルストーン達成報告、ROI分析提示 |
| コミュニケーション | 月次報告書（1ページサマリ）、重大判断時の臨時報告 |

### 4.2 SH-02: IT部門長（PO / CAB議長）

| 項目 | 内容 |
|------|------|
| 影響力 | 非常に高い（要件承認、優先度決定、CAB運営） |
| 関心度 | 非常に高い（プロジェクト成功が部門評価に直結） |
| 現在の態度 | 非常に支持的（プロジェクト発起人） |
| 期待 | 変更管理プロセスの完全電子化、監査工数削減 |
| 懸念 | 開発リソースの確保、他部門の協力度 |
| 管理戦略 | 隔週報告、スプリントレビューへの招待、意思決定の迅速化支援 |
| コミュニケーション | 隔週定例報告、Teamsチャット（随時）、緊急時は電話 |

### 4.3 SH-07〜SH-10: 各部門代表（CABメンバー/変更申請者）

| 項目 | 内容 |
|------|------|
| 影響力 | 中程度（変更要求の発信元、CAB投票権） |
| 関心度 | 高い（自部門業務への影響に敏感） |
| 現在の態度 | 中立〜懐疑的（新システム導入への不安あり） |
| 期待 | 操作が簡単で負担が増えないこと、変更の事前通知 |
| 懸念 | 業務負荷の増加、使いにくいシステム |
| 管理戦略 | UAT参加、トレーニング実施、フィードバック反映 |
| コミュニケーション | マイルストーン報告（メール）、トレーニング案内、FAQ |

### 4.4 SH-12: ISO20000審査機関

| 項目 | 内容 |
|------|------|
| 影響力 | 低い（プロジェクト自体への影響は限定的） |
| 関心度 | 高い（変更管理プロセスの準拠性を評価） |
| 現在の態度 | 中立（客観的な評価者） |
| 期待 | ISO20000要求事項を満たす証跡の提示 |
| 管理戦略 | 監査前の事前打合せ、証跡の整備 |
| コミュニケーション | 監査スケジュールに合わせた証跡提出 |

## 5. コミュニケーション計画

### 5.1 定例コミュニケーション

| 会議体/手段 | 頻度 | 対象者 | 形式 | 内容 | 責任者 |
|------------|------|--------|------|------|--------|
| 経営報告 | 月次 | SH-01 | 報告書（1ページ） | 進捗サマリ、KPI、リスク、次月計画 | PM |
| ステークホルダー定例 | 隔週（金曜） | SH-02, SH-03, SH-07〜10 | Teams会議（30分） | 進捗報告、デモ、フィードバック収集 | PM |
| スプリントレビュー | 隔週（金曜） | SH-02, SH-03, SH-04 | Teams会議（60分） | 成果物デモ、次スプリント計画 | PM |
| 開発チーム朝会 | 毎日 | SH-04 | Teamsスタンドアップ（15分） | 進捗共有、課題共有 | PM |
| 全社通知（変更管理関連） | 随時 | SH-11 | メール/Teams | 新システム案内、変更通知 | PM |

### 5.2 マイルストーン報告

| マイルストーン | 報告対象 | 報告形式 | 主な内容 |
|-------------|---------|---------|---------|
| M1: 計画完了 | SH-01, SH-02 | 報告書 + 会議 | 計画書承認依頼、スケジュール確認 |
| M2: 基盤構築完了 | SH-02, SH-03 | デモ + 報告 | 認証動作、基本画面デモ |
| M3: MVPリリース | SH-01, SH-02, SH-07〜10 | デモ + 説明会 | RFC→CAB承認フローデモ、利用開始案内 |
| M4: 全機能リリース | SH-01, SH-02, SH-07〜10 | デモ + 報告 | 全機能デモ、トレーニング計画案内 |
| M5: 本番運用開始 | 全員 | 全社メール + 説明会 | 利用開始案内、マニュアル配布 |
| M6: ISO20000監査 | SH-01, SH-02, SH-12 | 監査結果報告 | 監査結果、是正計画（該当する場合） |

### 5.3 エスカレーションルート

```
課題発生
  │
  ▼ 担当者レベルで解決可能か？
  ├─ YES → 担当者が対応・記録
  │
  └─ NO → PM にエスカレーション
            │
            ▼ PMレベルで解決可能か？
            ├─ YES → PM が対応・記録
            │
            └─ NO → PO（IT部門長）にエスカレーション
                      │
                      ▼ POレベルで解決可能か？
                      ├─ YES → PO が対応・記録
                      │
                      └─ NO → 経営層（スポンサー）にエスカレーション
```

### 5.4 コミュニケーション手段

| 手段 | 用途 | 対象 |
|------|------|------|
| Microsoft Teams チャネル | 日常的な開発コミュニケーション | 開発チーム |
| Microsoft Teams 会議 | 定例会議、レビュー | 全ステークホルダー |
| メール | 公式報告、承認依頼 | 全ステークホルダー |
| GitHub Issues / PR | 技術的な議論、コードレビュー | 開発チーム |
| SharePoint | ドキュメント共有 | 全ステークホルダー |
| 対面/電話 | 緊急時、機微な相談 | 必要に応じて |

## 6. ステークホルダーエンゲージメント計画

### 6.1 フェーズ別エンゲージメント

| フェーズ | 対象者 | エンゲージメント活動 | 目的 |
|---------|--------|-------------------|------|
| Phase 0 | SH-01, SH-02 | 計画承認、キックオフ参加 | 方針合意、コミットメント確保 |
| Phase 0 | SH-07〜10 | 要件ヒアリング | 現場ニーズの把握 |
| Phase 1 | SH-04, SH-05 | 技術レビュー参加 | 技術方針の合意 |
| Phase 2 | SH-02, SH-07〜10 | MVPデモ、フィードバック | 機能の妥当性確認 |
| Phase 3 | SH-06 | セキュリティレビュー | セキュリティ要件の充足確認 |
| Phase 4 | SH-07〜10 | トレーニング、UAT | 利用準備、品質確認 |
| Phase 4 | SH-11 | 全社アナウンス | 変更管理プロセス周知 |
| 安定化 | SH-12 | 監査対応 | ISO20000準拠証明 |

### 6.2 抵抗管理計画

| 想定される抵抗 | 対象者 | 原因 | 対応策 |
|-------------|--------|------|--------|
| 「今のやり方で問題ない」 | SH-07〜10 | 現状維持バイアス | ISO20000要件の説明、監査リスクの共有 |
| 「作業が増える」 | SH-07〜10 | 負荷増加の懸念 | 操作デモ（5分で起票完了）、自動化メリットの訴求 |
| 「システムが使いにくい」 | SH-07〜10 | UI/UXへの不満 | ユーザビリティテスト実施、フィードバック反映 |
| 「開発に時間が割けない」 | SH-04 | 兼務による負荷 | PO経由での業務調整、優先度の明確化 |

## 7. ステークホルダー満足度測定

### 7.1 測定方法

| 時期 | 対象 | 方法 | 質問項目 |
|------|------|------|---------|
| Phase 2完了後 | SH-02, SH-07〜10 | アンケート（5段階） | 機能充足度、操作性、期待との合致度 |
| 本番運用1ヶ月後 | SH-02, SH-07〜10, SH-11 | アンケート + ヒアリング | 利用頻度、操作性、改善要望 |
| 本番運用3ヶ月後 | 全内部SH | アンケート | 総合満足度、業務効率化実感 |

### 7.2 満足度指標

| 指標 | 目標値 | 測定タイミング |
|------|--------|-------------|
| システム利用率（アクティブユーザ/全対象者） | 80%以上 | 運用開始3ヶ月後 |
| ユーザ満足度（5段階平均） | 3.5以上 | 運用開始3ヶ月後 |
| RFC起票率（変更実施数に対するRFC起票率） | 100% | 運用開始3ヶ月後 |
| トレーニング受講率 | 100% | 本番稼働前 |
