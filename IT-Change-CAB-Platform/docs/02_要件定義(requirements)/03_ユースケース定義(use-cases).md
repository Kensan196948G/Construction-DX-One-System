## 1. アクター定義

### 1.1 主要アクター

| アクターID | アクター名 | 説明 | 組織 |
|-----------|----------|------|------|

## 3. RFC管理ユースケース

### UC-01: RFCを起票する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-01 |
| **ユースケース名** | RFCを起票する |
| **主アクター** | 申請者 |
| **事前条件** | 申請者がログイン済みであること |
| **事後条件** | RFCが作成され、RFC番号が採番されていること |
| **関連要件** | RFC-001 |

#### 基本フロー

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | RFC起票画面にアクセスする |
| 2 | システム | 起票フォームを表示する |
| 3 | 申請者 | 変更タイトルと説明を入力する |
| 4 | 申請者 | 変更種別（Emergency/Standard/Normal/Major）を選択する |
| 5 | システム | 変更種別に応じた入力項目を動的表示する |
| 6 | 申請者 | 対象システムを選択する |
| 7 | システム | 影響分析テンプレートを自動適用し、依存システムを表示する |
| 8 | 申請者 | 計画開始日時・終了日時を入力する |
| 9 | システム | 衝突検知を実行し、結果を表示する |
| 10 | 申請者 | 影響分析情報（業務影響・技術リスク・リスク評価）を入力する |
| 11 | 申請者 | ロールバック計画を入力する |
| 12 | 申請者 | 「下書き保存」または「提出」を選択する |
| 13 | システム | バリデーションを実行する |
| 14 | システム | RFC番号（RFC-YYYY-NNN）を自動採番し、保存する |
| 15 | システム | 提出の場合、承認ワークフローを開始し、通知を送信する |

#### 代替フロー

| 条件 | フロー |
|------|--------|
| A1: バリデーションエラー | ステップ13でエラーがあった場合、エラーメッセージを表示し、申請者に修正を求める |
| A2: 下書き保存 | ステップ12で下書き保存を選択した場合、ステータスをdraftとして保存し、後から編集可能とする |
| A3: フリーズ期間衝突 | ステップ9でBLOCKING衝突が検知された場合、警告を表示し、日程変更を促す |
| A4: 変更種別自動推奨 | ステップ6で対象システム選択後、システムが変更種別を自動推奨する |

### UC-03: RFCを提出する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-03 |
| **ユースケース名** | RFCを提出する |
| **主アクター** | 申請者 |
| **事前条件** | RFCがdraft状態で全必須項目が入力済みであること |
| **事後条件** | RFCがsubmitted状態に遷移し、承認ワークフローが開始されていること |
| **関連要件** | RFC-001, RFC-002, RFC-006 |

#### 基本フロー

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | RFC詳細画面で「提出」ボタンを押下する |
| 2 | システム | 全必須項目の入力チェックを実行する |
| 3 | システム | 変更種別の自動分類を実行し、推奨結果を表示する |
| 4 | システム | 衝突検知を最終実行する |
| 5 | システム | BLOCKING衝突がないことを確認する |
| 6 | システム | RFCステータスをsubmittedに変更する |
| 7 | システム | Standard変更の場合は自動承認し、approvedに遷移する |
| 8 | システム | Normal/Major変更の場合はCAB審議キューに追加する |
| 9 | システム | Emergency変更の場合はECABフローを開始する |
| 10 | システム | 関係者に通知を送信する |

## 4. CAB管理ユースケース

### UC-08: CAB会議をスケジュールする

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-08 |
| **ユースケース名** | CAB会議をスケジュールする |
| **主アクター** | CAB議長 / システム（自動） |
| **事前条件** | CABスケジュール設定が完了していること |
| **事後条件** | CAB会議がスケジュールされ、メンバーに通知されていること |
| **関連要件** | CAB-001 |

#### 基本フロー

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | システム | 毎週の定期CAB会議を自動スケジュールする |
| 2 | システム | 審議対象RFC（submitted状態）を収集する |
| 3 | システム | 審議対象が0件の場合はCAB議長に確認通知を送信する |
| 4 | システム | 3営業日前にCABメンバーに会議招集通知を送信する |
| 5 | システム | 前日にリマインダーと議題資料を送信する |

#### 代替フロー

| 条件 | フロー |
|------|--------|
| A1: 臨時CAB | CAB議長が手動で臨時CAB会議を追加スケジュールする |
| A2: 日程変更 | CAB議長が定期会議の日時を変更し、再通知する |
| A3: 審議案件なし | CAB議長が会議中止を判断し、キャンセル通知を送信する |

### UC-10: RFCを承認/却下する

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-10 |
| **ユースケース名** | RFCを承認/却下する |
| **主アクター** | CABメンバー / CAB議長 / 経営承認者 |
| **事前条件** | RFCがcab_review状態であること |
| **事後条件** | 投票が記録され、承認条件達成時はステータスが更新されていること |
| **関連要件** | CAB-003, CAB-004 |

#### 基本フロー

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | CABメンバー | RFC詳細画面またはCABダッシュボードから承認操作画面を表示する |
| 2 | システム | RFC情報、影響分析、衝突検知結果を表示する |
| 3 | CABメンバー | 承認アクションを選択する（承認/条件付き承認/却下/保留/差戻し） |
| 4 | CABメンバー | コメントを入力する（却下・保留・差戻し時は必須） |
| 5 | CABメンバー | 投票を確定する |
| 6 | システム | 投票を記録する |
| 7 | システム | 承認条件を判定する（最低承認人数、必須承認者） |
| 8 | システム | 条件達成時はRFCステータスをapprovedに変更する |
| 9 | システム | 申請者に審議結果を通知する |

#### 代替フロー

| 条件 | フロー |
|------|--------|
| A1: Major変更 | 経営承認者の承認も必須。経営承認者に追加通知を送信する |
| A2: 条件付き承認 | 条件をRFCに記録し、申請者に条件充足を求める |
| A3: 承認期限超過 | エスカレーション通知をCAB議長に送信する |
| A4: 差し戻し | RFCをdraft状態に戻し、申請者に修正を依頼する |
