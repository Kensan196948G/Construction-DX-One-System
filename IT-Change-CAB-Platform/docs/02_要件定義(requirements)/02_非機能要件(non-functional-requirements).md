## 1. 文書概要

本文書は、IT変更管理・リリース自動化プラットフォームの非機能要件を定義する。性能、可用性、セキュリティ、拡張性、保守性、運用性の各観点から要件を規定する。

---

## 2. 性能要件

### 2.1 応答時間要件

| 操作 | 応答時間要件 | 測定条件 | 優先度 |
|------|------------|---------|--------|
| RFC一覧表示 | 2秒以内 | 1000件のRFCデータ、30名同時アクセス | 必須 |
| RFC起票・保存 | 2秒以内 | 通常サイズのRFC（添付ファイル除く） | 必須 |
| RFC詳細表示 | 2秒以内 | 関連データ（承認履歴・コメント含む） | 必須 |
| 承認操作 | 1秒以内 | 単一承認アクション | 必須 |
| 承認通知送信 | 1分以内 | メール・Teams通知の送信完了 | 必須 |
| 変更カレンダー表示 | 2秒以内 | 月間ビュー、100件の変更データ | 必須 |
| KPIダッシュボード表示 | 3秒以内 | 全KPIの集計・表示 | 必須 |
| 影響分析実行 | 3秒以内 | 依存関係探索・リスク計算 | 必須 |
| 衝突検知実行 | 2秒以内 | 同時期変更・フリーズ期間チェック | 必須 |
| 検索・フィルタ | 2秒以内 | 全文検索・条件フィルタリング | 必須 |
| レポート生成（PDF） | 10秒以内 | 月次KPIレポート生成 | 推奨 |
| CSVエクスポート | 5秒以内 | 1000件以下のデータエクスポート | 推奨 |

### 2.2 スループット要件

| 項目 | 要件値 | 説明 |
|------|--------|------|
| 同時利用ユーザ数 | 30名 | ピーク時の同時アクセスユーザ数 |
| 同時API要求数 | 100リクエスト/秒 | API全体の処理能力 |
| バッチ通知送信 | 100件/分 | 一斉通知送信能力 |
| ファイルアップロード | 50MB/ファイル | 添付ファイルの最大サイズ |

### 2.3 データ容量要件

| 項目 | 想定容量 | 保持期間 |
|------|---------|---------|
| RFC年間件数 | 500件/年 | 永久保存 |
| 添付ファイル | 50GB/年 | 5年間 |
| 監査ログ | 10GB/年 | 7年間 |
| バックアップ | 100GB | 3世代保持 |

## 3. 可用性要件

### 3.1 稼働率要件

| 項目 | 要件値 | 説明 |
|------|--------|------|
| 年間稼働率 | 99.5%以上 | 計画停止除く |
| 月間最大停止時間 | 3.6時間以内 | 計画外停止 |
| 年間最大停止時間 | 43.8時間以内 | 計画外停止 |
| 計画停止通知 | 5営業日前 | メンテナンスウィンドウ |

### 3.2 緊急変更対応

| 項目 | 要件値 | 説明 |
|------|--------|------|
| 緊急変更受付 | 24時間365日 | Emergency変更の起票・承認機能 |
| ECAB承認 | 24時間365日 | 緊急CAB承認の常時利用可能 |
| 緊急通知 | 24時間365日 | Teams・メール通知の常時送信 |

### 3.3 災害復旧要件

| 項目 | 要件値 | 説明 |
|------|--------|------|
| RTO（目標復旧時間） | 4時間以内 | サービス復旧までの時間 |
| RPO（目標復旧時点） | 1時間以内 | データ損失許容時間 |
| バックアップ頻度 | 日次フルバックアップ | PostgreSQLデータベース |
| WALアーカイブ | 継続的 | ポイントインタイムリカバリ用 |
| バックアップ保持 | 3世代（3日分） | ローテーション管理 |
| DR訓練 | 年2回 | 復旧手順の定期検証 |

### 3.4 冗長化要件

| コンポーネント | 冗長化方式 | 説明 |
|-------------|-----------|------|
| アプリケーションサーバ | マルチプロセス（PM2） | プロセス異常時の自動再起動 |

| Redis | Sentinel構成 | マスター/スレーブ自動フェイルオーバー |
| ファイルストレージ | RAID構成 | 添付ファイルの冗長保存 |

## 4. セキュリティ要件

### 4.1 認証・認可

| 項目 | 要件 | 説明 |
|------|------|------|
| 認証方式 | JWT + セッション管理 | アクセストークン + リフレッシュトークン |
| パスワードポリシー | 12文字以上、大小英数記号必須 | bcryptハッシュ保存 |
| 多要素認証（MFA） | TOTP対応（推奨） | 管理者アカウントは必須 |
| セッションタイムアウト | 30分（操作なし時） | セキュリティポリシー準拠 |
| アカウントロック | 5回連続失敗でロック | 30分後自動解除 |
| SSO連携 | Entra ID SAML/OIDC対応 | 既存認証基盤との統合 |

### 4.2 権限管理（RBAC）

| ロール | 説明 | 主要権限 |
|--------|------|---------|
| admin | システム管理者 | 全機能アクセス、ユーザ管理、システム設定 |
| cab_chair | CAB議長 | CAB管理、全RFC承認、フリーズ期間管理 |
| cab_member | CABメンバー | RFC承認・却下、議事録閲覧 |
| requester | 申請者 | RFC起票・編集、自身のRFC閲覧 |
| implementer | 実施者 | 変更実施記録、チェックリスト操作 |
| viewer | 閲覧者 | RFC・カレンダー・KPIの閲覧のみ |
| executive | 経営承認者 | Major変更の承認、KPIダッシュボード閲覧 |

### 4.3 データ保護

| 項目 | 要件 | 説明 |
|------|------|------|
| 通信暗号化 | TLS 1.2以上必須 | 全通信のHTTPS化 |
| データベース暗号化 | AES-256 | 保存時暗号化（at-rest encryption） |
| 個人情報マスキング | ログ出力時 | ユーザ名・メールアドレスのマスキング |
| API認証 | Bearer Token | 全APIエンドポイントで認証必須 |
| CORS設定 | ホワイトリスト方式 | 許可オリジンの明示的設定 |
| CSRF対策 | CSRFトークン | フォーム送信時の検証 |
| XSS対策 | 入力サニタイズ + CSP | コンテンツセキュリティポリシー設定 |
| SQLインジェクション対策 | パラメータバインド | ORMによるクエリ構築 |

### 4.4 監査ログ

| ログ種別 | 記録内容 | 保持期間 |
|---------|---------|---------|
| 認証ログ | ログイン/ログアウト、認証失敗 | 7年 |
| 操作ログ | RFC操作、承認操作、設定変更 | 7年 |
| アクセスログ | API呼び出し、画面アクセス | 3年 |
| エラーログ | アプリケーションエラー、システムエラー | 1年 |
| 変更履歴 | データ変更のBefore/After | 7年（ISO20000準拠） |

### 4.5 セキュリティ監査

| 項目 | 頻度 | 説明 |
|------|------|------|
| 脆弱性スキャン | 月次 | 依存パッケージの脆弱性チェック（npm audit） |
| ペネトレーションテスト | 年次 | 外部専門業者による侵入テスト |
| セキュリティレビュー | コードレビュー時 | PRマージ前のセキュリティチェック |
| アクセス権限棚卸 | 四半期 | ユーザ権限の見直し |

## 6. 保守性要件

| 項目 | 基準 | ツール |

| コードカバレッジ | 80%以上 | Jest（バックエンド）、Vitest（フロントエンド） |
| 静的解析 | エラー0件 | ESLint（TypeScript strict） |
| コードフォーマット | 統一 | Prettier |
| 型安全性 | strict mode | TypeScript strict: true |
| 循環参照 | 0件 | madge |
| コード複雑度 | サイクロマティック複雑度10以下 | ESLint complexity rule |

### 6.2 ドキュメント

| 項目 | 要件 | 説明 |

| 運用手順書 | Markdown | デプロイ・障害対応・バックアップ手順 |

### 6.3 テスト

| セキュリティテスト | 脆弱性0件 | npm audit / OWASP ZAP | 週次 |

### 6.4 CI/CDパイプライン

| ステージ | 内容 | トリガー |
