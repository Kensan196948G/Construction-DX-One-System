## 1. 文書概要

本文書は、IT変更管理・リリース自動化プラットフォームにおける全機能要件を定義する。ISO20000変更管理プロセスに準拠し、RFC（Request for Change）の起票からCAB（Change Advisory Board）承認、変更実施、PIR（Post Implementation Review）までの全プロセスを網羅する。

### 1.1 準拠規格

| 規格 | 対象条項 | 概要 |
|------|----------|------|
| ISO20000-1:2018 | 変更管理プロセス | 変更管理の計画・制御・承認・実施・レビュー |
| ISO27001:2022 | A.8.32 変更管理 | セキュリティを考慮した変更手順 |
| ITIL v4 | 変更管理プラクティス | 変更管理ガイドライン |

### 1.2 対象システム

| システム名 | 分類 | 影響ユーザ規模 |
|-----------|------|---------------|
| Active Directory | オンプレミス認証基盤 | 全社員 |
| Entra ID (Azure AD) | クラウド認証基盤 | 全社員 |
| HENGEONE | SSO/MFA基盤 | 全社員 |
| Microsoft 365 | クラウドグループウェア | 全社員 |
| Exchange Online | メール基盤 | 全社員 |
| DeskNet'sNeo | グループウェア | 全社員 |
| AppSuite | ワークフロー基盤 | 全社員 |
| ファイルサーバ | オンプレミスストレージ | 全社員 |
| ネットワーク機器 | インフラ基盤 | 全社員 |
| Azure リソース | クラウドインフラ | IT部門 |

## 2. RFC（変更申請）管理

### 2.1 RFC-001: RFC起票フォーム

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-001 |
| **要件名** | RFC起票フォーム |
| **優先度** | 必須（P1） |
| **カテゴリ** | RFC管理 |

#### 機能概要

変更申請者が変更内容をWebフォームから起票できる機能。変更種別・対象システム・影響分析・ロールバック計画を含む包括的な申請フォームを提供する。

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | RFC起票画面にアクセスする |
| 2 | システム | 起票フォームを表示する（変更種別選択、対象システム選択、日程入力等） |
| 3 | 申請者 | 必須項目を入力する（タイトル、説明、変更種別、対象システム、計画日時） |
| 4 | 申請者 | 影響分析情報を入力する（業務影響、技術リスク、影響ユーザ数） |
| 5 | 申請者 | ロールバック計画を入力する（手順、判断基準、所要時間） |
| 6 | 申請者 | 下書き保存または提出を行う |
| 7 | システム | 入力バリデーションを実行し、RFC番号を自動採番する（RFC-YYYY-NNN形式） |
| 8 | システム | 提出時は承認ワークフローを開始し、関係者に通知する |

#### 入力項目

| 項目名 | 型 | 必須 | 説明 |
|--------|-----|------|------|
| タイトル | テキスト(300文字) | 必須 | 変更の概要タイトル |
| 説明 | テキスト(無制限) | 必須 | 変更の詳細説明 |
| 変更種別 | 選択 | 必須 | Emergency/Standard/Normal/Major |
| 優先度 | 選択 | 必須 | Low/Medium/High/Critical |
| 対象システム | 複数選択 | 必須 | 変更対象のシステム群 |
| 影響ユーザ数 | 数値 | 必須 | 影響を受けるユーザの推定人数 |
| 計画開始日時 | 日時 | 必須 | 変更実施の計画開始日時 |
| 計画終了日時 | 日時 | 必須 | 変更実施の計画終了日時 |
| 業務影響 | テキスト | 必須 | 業務への影響内容 |
| 技術リスク | 選択 | 必須 | Low/Medium/High/Critical |
| リスク評価 | テキスト | 必須 | リスクの詳細評価 |
| ロールバック計画 | テキスト | 必須 | ロールバック手順と判断基準 |
| ロールバック判断基準 | テキスト | 任意 | ロールバック開始の判断条件 |
| 関連インシデントID | UUID | 任意 | 緊急変更の場合の関連インシデント |
| 添付ファイル | ファイル | 任意 | 手順書・設計書等の添付 |

#### 受け入れ基準

- [ ] 全必須項目が入力されないと提出できないこと
- [ ] RFC番号がRFC-YYYY-NNN形式で自動採番されること
- [ ] 下書き保存ができ、後から編集・提出できること
- [ ] 提出後にステータスが「submitted」に変更されること
- [ ] 提出時に承認者・関係者へ通知が送信されること
- [ ] 変更種別に応じた必須項目の動的制御が行われること

### 2.2 RFC-002: 変更種別自動分類

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-002 |
| **要件名** | 変更種別自動分類 |
| **優先度** | 必須（P1） |
| **カテゴリ** | RFC管理 |

#### 機能概要

変更内容・対象システム・影響範囲に基づき、変更種別（Standard/Normal/Major/Emergency）を自動判定し、申請者に推奨する機能。

#### 自動分類ルール

| 条件 | 判定結果 | 承認レベル |
|------|---------|-----------|
| 事前承認済みテンプレートに該当 | Standard（標準変更） | 自動承認 |
| 関連インシデントID指定あり + Critical優先度 | Emergency（緊急変更） | ECAB承認 |
| コアシステム（AD/EntraID/HENGEONE）を含む + 影響ユーザ100人以上 | Major（重大変更） | 経営層+CAB承認 |
| 上記いずれにも該当しない | Normal（通常変更） | CAB承認 |

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | 対象システムと変更内容を入力する |
| 2 | システム | 入力内容に基づき変更種別を自動判定する |
| 3 | システム | 推奨変更種別を表示し、判定理由を提示する |
| 4 | 申請者 | 推奨に同意するか、手動で変更種別を変更する |
| 5 | システム | 手動変更の場合は変更理由の入力を求める |

#### 受け入れ基準

- [ ] 対象システム・影響範囲入力時に自動分類が実行されること
- [ ] 分類理由が申請者に表示されること
- [ ] 手動で変更種別を上書きできること（理由入力必須）
- [ ] 標準変更テンプレートが管理画面から登録・編集できること

### 2.3 RFC-003: 影響分析テンプレート

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-003 |
| **要件名** | 影響分析テンプレート |
| **優先度** | 必須（P1） |
| **カテゴリ** | RFC管理 |

#### 機能概要

依存システム・ユーザ影響・リスク評価を標準化されたテンプレートに基づいて分析する機能。システム依存関係マップを用いた自動分析と、手動入力を組み合わせる。

#### 依存関係マップ

| 対象システム | 依存先システム |
|-------------|--------------|
| Active Directory | EntraID, HENGEONE, FileServer, DeskNetsNeo |
| Entra ID | EntraID Connect, HENGEONE, Microsoft365, Exchange Online |
| HENGEONE | Microsoft365, DeskNetsNeo, AppSuite |
| EntraID Connect | Active Directory, Entra ID |
| Exchange Online | Entra ID, HENGEONE |
| ファイルサーバ | Active Directory |
| DeskNet'sNeo | Active Directory, HENGEONE |
| AppSuite | Active Directory, HENGEONE |

#### リスクスコアリング

| 評価項目 | 配点 | 説明 |
|---------|------|------|
| 影響システム数 | システム数 x 2点 | 直接・間接影響システムの合計 |
| コアシステム影響 | 該当数 x 5点 | AD/EntraID/HENGEONEへの影響 |
| 変更種別スコア | 0-8点 | Standard:0, Normal:3, Emergency:5, Major:8 |
| **合計20点以上** | **Critical** | 経営層承認必須 |
| **合計12-19点** | **High** | CAB全メンバー承認必須 |
| **合計6-11点** | **Medium** | CAB通常審議 |
| **合計5点以下** | **Low** | 簡易承認可 |

#### 受け入れ基準

- [ ] 対象システム選択時に依存関係が自動表示されること
- [ ] リスクスコアが自動計算されること
- [ ] 影響分析結果がRFCに自動添付されること
- [ ] 影響分析テンプレートが管理画面からカスタマイズできること

### 2.4 RFC-004: ロールバック計画必須化

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-004 |
| **要件名** | ロールバック計画必須化 |
| **優先度** | 必須（P1） |
| **カテゴリ** | RFC管理 |

#### 機能概要

通常変更・重大変更において、ロールバック計画（手順、判断基準、推定所要時間）の記載を必須とする機能。標準変更は任意。

#### ロールバック計画の必須項目

| 項目 | Standard | Normal | Major | Emergency |
|------|----------|--------|-------|-----------|
| ロールバック手順 | 任意 | 必須 | 必須 | 事後記録可 |
| 判断基準 | 任意 | 必須 | 必須 | 事後記録可 |
| 推定所要時間 | 任意 | 必須 | 必須 | 事後記録可 |
| 検証手順 | 任意 | 推奨 | 必須 | 事後記録可 |
| 連絡先一覧 | 任意 | 推奨 | 必須 | 必須 |

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | 変更種別をNormalまたはMajorに設定する |
| 2 | システム | ロールバック計画の入力欄を必須表示する |
| 3 | 申請者 | ロールバック手順、判断基準、推定時間を入力する |
| 4 | システム | ロールバック計画の完全性をバリデーションする |
| 5 | システム | ロールバック計画未入力の場合はRFC提出をブロックする |

#### 受け入れ基準

- [ ] Normal/Major変更でロールバック計画が未入力の場合、提出できないこと
- [ ] Emergency変更では事後入力が許可されること
- [ ] Standard変更ではロールバック計画が任意であること
- [ ] ロールバック計画の完全性チェックが自動で行われること

### 2.5 RFC-005: CMDB自動連携

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-005 |
| **要件名** | CMDB自動連携 |
| **優先度** | 推奨（P2） |

#### 機能概要

変更対象のCI（Configuration Item）をCMDB（構成管理データベース）から自動取得し、影響を受けるCIを特定する機能。

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | 対象システムを選択する |
| 2 | システム | CMDBから該当CIの一覧を取得・表示する |
| 3 | 申請者 | 変更対象CIを選択する |
| 4 | システム | 選択CIの依存関係ツリーを表示する |
| 5 | システム | 影響を受けるCIを自動特定し、RFCに添付する |

#### 受け入れ基準

- [ ] CMDBからCIリストが取得できること
- [ ] CI間の依存関係ツリーが表示されること
- [ ] 影響CIが自動特定されRFCに記録されること
- [ ] CMDB未連携時は手動入力にフォールバックすること

### 2.6 RFC-006: 変更衝突検知

| 項目 | 内容 |
|------|------|
| **要件ID** | RFC-006 |
| **要件名** | 変更衝突検知 |
| **優先度** | 必須（P1） |
| **カテゴリ** | RFC管理 |

#### 機能概要

同一システムへの重複変更、フリーズ期間中の変更、依存システムの同時変更を自動検知し、申請者・CABに警告する機能。

#### 衝突検知ルール

| 検知タイプ | 重大度 | 説明 | アクション |
|-----------|--------|------|-----------|
| 同一システム同時変更 | HIGH | 同一対象システムへの変更が同一期間にスケジュール | 警告表示、CABに通知 |
| フリーズ期間違反 | BLOCKING | 変更禁止期間中の変更申請 | 提出ブロック（Emergency除く） |
| 依存システム変更 | MEDIUM | 依存先システムの同時期変更 | 注意喚起表示 |
| 連続変更検知 | LOW | 同一システムへの短期間連続変更 | 情報表示 |

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | 申請者 | 対象システムと計画日時を入力する |
| 2 | システム | リアルタイムで衝突チェックを実行する |
| 3 | システム | 衝突・警告がある場合はメッセージを表示する |
| 4 | 申請者 | 衝突内容を確認し、日程変更または続行を判断する |
| 5 | システム | BLOCKING衝突がある場合は提出をブロックする |
| 6 | システム | HIGH/MEDIUM衝突はCAB審議資料に自動添付する |

#### 受け入れ基準

- [ ] 同一システム・同一期間の変更が検知されること
- [ ] フリーズ期間中の変更が検知されブロックされること（Emergency除外）
- [ ] 依存システムの同時変更が検知されること
- [ ] 衝突情報がCAB審議資料に自動添付されること
- [ ] 検知結果がリアルタイムで表示されること

## 3. CAB（変更諮問委員会）管理

### 3.1 CAB-001: CAB会議スケジュール管理

| 項目 | 内容 |
|------|------|
| **要件ID** | CAB-001 |
| **要件名** | CAB会議スケジュール管理 |
| **優先度** | 必須（P1） |
| **カテゴリ** | CAB管理 |

#### 機能概要

定期CAB会議（週次）の自動スケジュール・召集・議題配布を行う機能。CABメンバーへのリマインダー通知も含む。

#### CAB構成

| 役割 | メンバー | 承認権限 | 出席要件 |
|------|---------|---------|---------|
| CAB議長 | IT部門長 | 全変更種別 | 必須 |
| 技術メンバー | IT部門員（3名） | 通常変更 | 定足数2名以上 |
| ビジネスメンバー | 各部門代表 | 重大変更（影響部門） | Major時必須 |
| セキュリティレビュア | IT部門セキュリティ担当 | セキュリティ影響変更 | セキュリティ変更時必須 |
| 経営承認者 | 経営層 | 重大変更のみ | Major時必須 |

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | システム | 毎週固定日時にCAB会議を自動スケジュールする |
| 2 | システム | 審議対象RFC（submitted状態）から議題を自動生成する |
| 3 | システム | CABメンバーに会議招集通知を送信する（3営業日前） |
| 4 | システム | 会議前日にリマインダーと議題資料を送信する |
| 5 | CAB議長 | 会議日時の変更・臨時CABの追加設定を行う |
| 6 | システム | 会議結果の記録と議事録の自動生成を行う |

#### 受け入れ基準

- [ ] 週次CAB会議が自動でスケジュールされること
- [ ] 審議対象RFCが自動で議題にリストアップされること
- [ ] 会議招集通知が3営業日前に送信されること
- [ ] 前日リマインダーが送信されること
- [ ] 臨時CAB会議が手動で追加設定できること
- [ ] Teams連携で会議リンクが自動生成されること

### 3.2 CAB-002: 審議議題自動生成

| 項目 | 内容 |
|------|------|
| **要件ID** | CAB-002 |
| **要件名** | 審議議題自動生成 |
| **優先度** | 必須（P1） |
| **カテゴリ** | CAB管理 |

#### 機能概要

承認待ち状態のRFCから、CAB会議の審議議題を自動生成する。議題にはRFC概要、影響分析結果、リスク評価、衝突検知結果を含む。

#### 議題テンプレート構成

| セクション | 内容 | 自動生成 |
|-----------|------|---------|
| 議題ヘッダ | 会議日時、出席予定メンバー | 自動 |
| 前回会議フォローアップ | 前回deferred/保留RFCの状況 | 自動 |
| 新規審議案件 | submitted状態のRFC一覧（リスク降順） | 自動 |
| 各RFC詳細 | 影響分析・リスク評価・衝突検知結果 | 自動 |
| 変更カレンダー | 当週〜来週の変更予定一覧 | 自動 |
| その他連絡事項 | 手動追加事項 | 手動 |

#### 受け入れ基準

- [ ] submitted状態のRFCが自動で議題に含まれること
- [ ] リスクレベルの高い順にRFCがソートされること
- [ ] 影響分析結果・衝突検知結果が議題に含まれること
- [ ] 前回保留案件が自動で再掲載されること
- [ ] 議題がPDF/HTML形式で出力できること

### 3.3 CAB-003: オンライン承認機能

| 項目 | 内容 |
|------|------|
| **要件ID** | CAB-003 |
| **要件名** | オンライン承認機能 |
| **優先度** | 必須（P1） |
| **カテゴリ** | CAB管理 |

#### 機能概要

CABメンバーがWeb画面・メール・Teams経由でRFCを審議・承認・却下できる機能。対面会議に加え、非同期オンライン承認にも対応する。

#### 承認アクション

| アクション | 説明 | コメント | 対応チャネル |
|-----------|------|---------|-------------|
| 承認（Approve） | RFC実施を承認する | 任意 | Web/メール/Teams |
| 条件付き承認（Conditional） | 条件付きで承認する | 必須（条件記載） | Web/メール/Teams |
| 却下（Reject） | RFC実施を却下する | 必須（却下理由） | Web/メール/Teams |
| 保留（Defer） | 追加情報を要求し保留する | 必須（必要情報） | Web/メール/Teams |
| 差し戻し（Return） | 申請者に修正を求め差し戻す | 必須（修正指示） | Web |

#### ユースケース

| ステップ | アクター | アクション |
|---------|---------|----------|
| 1 | システム | CABメンバーに承認依頼通知を送信する（メール/Teams） |
| 2 | CABメンバー | 通知内のリンクからRFC詳細を確認する |
| 3 | CABメンバー | 影響分析・リスク評価・衝突検知結果を確認する |
| 4 | CABメンバー | 承認/却下/保留のアクションを実行する |
| 5 | システム | 投票結果を記録し、承認条件を判定する |
| 6 | システム | 承認条件達成時にRFCステータスを更新する |
| 7 | システム | 申請者に審議結果を通知する |

#### 受け入れ基準

- [ ] Web画面から承認・却下・保留操作ができること
