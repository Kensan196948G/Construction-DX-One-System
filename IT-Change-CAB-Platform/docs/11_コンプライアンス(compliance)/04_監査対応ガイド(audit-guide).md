## 1. 概要

### 1.1 目的

本文書は、IT-Change-CAB-Platformに関する監査（内部監査・外部監査）に対応するためのガイドである。監査時に提出すべき証跡の一覧、証跡の保存場所、取得手順、および監査対応の手順を定義する。

### 1.2 対象監査

| 監査種別 | 頻度 | 実施者 | 対象規格 |
|---------|------|--------|---------|
| ISO20000内部監査 | 年1回 | 内部監査チーム | ISO/IEC 20000-1:2018 |
| ISO20000外部監査（認証審査） | 年1回 | 認証機関 | ISO/IEC 20000-1:2018 |
| ISO27001内部監査 | 年1回 | 内部監査チーム | ISO/IEC 27001:2022 |
| ISO27001外部監査（認証審査） | 年1回 | 認証機関 | ISO/IEC 27001:2022 |
| IT統制監査（J-SOX） | 年1回 | 監査法人 | 金融商品取引法 |
| セキュリティ監査 | 年1回 | 外部セキュリティベンダー | NIST CSF / 社内ポリシー |

## 3. 証跡取得手順

### 3.1 システムからのレポート取得

#### 3.1.1 RFCレポート（変更申請一覧）

```
取得方法: レポートAPI
エンドポイント: GET /api/v1/reports/rfcs
パラメータ:
  - start_date: 対象期間開始日（YYYY-MM-DD）
  - end_date: 対象期間終了日（YYYY-MM-DD）
  - format: csv / pdf / json
権限: SYS_ADMIN
出力形式: CSV（Excel互換）/ PDF

出力項目:
  - RFC番号
  - タイトル
  - 変更種別
  - 優先度
  - ステータス
  - 対象システム
  - リスクレベル
  - 申請者
  - 実施者
  - 申請日
  - 承認日
  - 実施開始日
  - 実施完了日
  - 結果（成功/ロールバック/キャンセル）
```

#### 3.1.2 CAB審議レポート

```
取得方法: レポートAPI
エンドポイント: GET /api/v1/reports/cab-decisions
パラメータ:
  - start_date: 対象期間開始日
  - end_date: 対象期間終了日
  - format: csv / pdf
権限: SYS_ADMIN, CAB_CHAIR

出力項目:
  - RFC番号
  - CABセッション日時
  - 審議結果（承認/却下/差戻し）
  - 承認条件
  - 投票結果（賛成/反対数）
  - 決定者（議長）
  - 決定理由
```

#### 3.1.3 監査ログレポート

```
取得方法: 監査ログAPI
エンドポイント: GET /api/v1/admin/audit-logs
パラメータ:
  - start_date: 対象期間開始日
  - end_date: 対象期間終了日
  - event_type: イベント種別フィルタ（任意）
  - user_id: ユーザIDフィルタ（任意）
  - format: csv / json
権限: SYS_ADMIN

出力項目:
  - タイムスタンプ
  - イベント種別
  - ユーザID
  - ユーザロール
  - アクション
  - リソース種別
  - リソースID
  - 結果（成功/失敗）
  - 詳細
  - ソースIP
```

#### 3.1.4 KPIレポート

```
取得方法: KPI API
エンドポイント: GET /api/v1/reports/kpi
パラメータ:
  - period: 月次 / 四半期 / 年次
  - start_date: 対象期間開始日
  - end_date: 対象期間終了日
  - format: csv / pdf
権限: SYS_ADMIN, CAB_CHAIR, SEC_REVIEWER, EXEC_APPROVER

出力項目:
  - 変更成功率（%）
  - ロールバック率（%）
  - 変更起因インシデント率（%）
  - 平均承認リードタイム（時間）
  - 変更件数（種別別）
  - 変更件数（システム別）
  - 緊急変更件数
  - PIR実施率（%）
```

### 3.2 CSVエクスポート手順

1. SYS_ADMINロールのアカウントでシステムにログイン
2. 管理画面 > レポート > エクスポートを選択
3. 対象データ種別を選択（RFC/CAB決定/監査ログ/KPI）
4. 対象期間を指定
5. 出力形式（CSV/PDF）を選択
6. 「エクスポート」ボタンを実行
7. ダウンロードされたファイルを監査証跡フォルダに保存

### 3.3 長期アーカイブからの取得

5年以上前の証跡は、コールドストレージ（Azure Blob Storage）に保存されている。

```
取得手順:
1. SYS_ADMINが Azure Portal にログイン
2. ストレージアカウント > コンテナ > cab-platform-archive を選択
3. 対象期間のフォルダを選択（YYYY/MM/形式）
4. 必要なファイルをダウンロード
5. ファイルの整合性をハッシュ値で検証
```

## 4. 監査対応手順

### 4.1 監査準備フェーズ

| # | 作業 | 担当者 | 時期 | 成果物 |
|---|------|--------|------|--------|
| 1 | 監査スケジュール確認 | IT部門長 | 監査2ヶ月前 | 監査対応計画 |

| 3 | 証跡の事前準備・確認 | SYS_ADMIN | 監査1ヶ月前 | 証跡一覧チェックリスト |
| 4 | 証跡の取得・整理 | SYS_ADMIN | 監査2週間前 | 整理済み証跡ファイル |
| 5 | 証跡の完全性確認 | セキュリティレビュア | 監査1週間前 | 証跡確認報告書 |
| 6 | 監査対応者のブリーフィング | IT部門長 | 監査1週間前 | ブリーフィング資料 |
| 7 | 監査用アカウントの準備（必要時） | SYS_ADMIN | 監査3日前 | 閲覧専用アカウント |

### 4.2 監査実施フェーズ

| # | 作業 | 担当者 | 内容 |
|---|------|--------|------|
| 1 | 監査員の受入れ | IT部門長 | 監査員への挨拶、システム概要説明 |
| 2 | 証跡の提出 | SYS_ADMIN | 事前準備した証跡の提出 |
| 3 | システムデモンストレーション | SYS_ADMIN / CAB議長 | RFCワークフロー、CAB承認、KPIダッシュボードの実演 |

| 5 | 監査員のシステム確認支援 | SYS_ADMIN | 閲覧用アカウントでのシステム操作支援 |
| 6 | 指摘事項の確認 | IT部門長 | 監査員からの指摘事項の確認・理解 |
