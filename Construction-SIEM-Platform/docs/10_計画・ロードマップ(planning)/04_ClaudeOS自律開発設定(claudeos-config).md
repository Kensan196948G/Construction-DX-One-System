## ⏱ ランタイム設定

### 基本設定

| 項目 | 値 | 説明 |
|------|-----|------|
| 最大ランタイム | **8時間** | 強制終了タイマー |
| モード | Auto Mode | 自律開発 |
| エージェント | Agent Teams | 複数エージェント協調 |
| 並列開発 | WorkTree Parallel | Git worktree使用 |
| プロジェクト管理 | GitHub Projects | Issues/PR/Actions連動 |

### 停止条件

```
停止トリガー:
├── 8時間タイマー到達 → 強制終了
├── Loop Guard 発動 → 緊急停止
├── CTO判断 → 手動停止
└── STABLE達成 + 全タスク完了 → 正常終了
```

## 🔁 ループ構成

### 4フェーズループ

```
┌──────────────────────────────────────────┐
│            8時間ランタイム                  │
│                                          │
│  ┌────────┐  ┌────────┐  ┌────────┐     │
│  │Monitor │  │ Build  │  │ Verify │     │
│  │  1h    │─►│  2h    │─►│  2h    │     │
│  └────────┘  └────────┘  └───┬────┘     │
│                              │           │
│                    失敗 ◄────┤           │
│                    │         │ 成功      │
│              ┌─────▼─────┐   │           │
│              │CI Manager │   │           │
│              │Auto Repair│   │           │
│              └─────┬─────┘   │           │
│                    │         │           │
│                    ▼         ▼           │
│              再Verify   ┌────────┐      │
│                         │Improve │      │
│                         │  3h    │      │
│                         └────────┘      │
└──────────────────────────────────────────┘
```

### フェーズ詳細

| フェーズ | 時間 | 活動内容 | 成果物 |
|---------|:----:|---------|--------|
| 🔍 **Monitor** | 1時間 | GitHub Issues確認、Projects状態確認、優先タスク選定 | タスクリスト |
| 🏗 **Build** | 2時間 | 設計・実装、WorkTree並列開発 | コード変更 |
| ✅ **Verify** | 2時間 | テスト実行、CI確認、品質チェック | テスト結果 |
| 📈 **Improve** | 3時間 | リファクタリング、ドキュメント更新、次タスク準備 | PR作成 |

### 失敗時フロー

```
Verify 失敗
    │
    ▼
CI Manager 起動
    │
    ├── エラー分析
    │   ├── black/isort → 自動修正
    │   ├── flake8 → コード修正
    │   ├── pytest → テスト修正
    │   └── bandit → セキュリティ修正
    │
    ├── Auto Repair 実行
    │   ├── 自動修正コミット
    │   └── CI再実行
    │
    └── 再Verify
        ├── 成功 → Improve フェーズへ
        └── 失敗 → 最大3回リトライ → 手動対応必要
```

## ✅ STABLE判定

### 判定条件

```
STABLE = 以下すべてが成功:
├── ✅ test: 全テスト合格
├── ✅ CI: GitHub Actions 全ジョブ成功
├── ✅ lint: black + isort + flake8 パス
├── ✅ build: Docker ビルド成功
├── ✅ error: 0件
└── ✅ security: bandit 0 issues
```

### 連続成功回数 (N)

| 重要度 | N値 | 説明 |
|--------|:---:|------|
| 🟢 small | 2 | 軽微な変更（typo修正、ドキュメント） |
| 🟡 normal | **3** | 通常の機能追加・修正 |
| 🔴 critical | **5** | セキュリティ修正、コア機能変更 |

### 判定フロー

```
コミット → CI実行 → 結果記録

連続成功カウント:
  成功: カウント +1
  失敗: カウント = 0（リセット）

カウント >= N:
  → STABLE 判定
  → PR作成可能
  → マージ承認可能
```

## 📊 GitHub Projects連動

### ステータスフロー

```
Inbox → Backlog → Ready → Development → Verify → Deploy Gate → Done
                                                       │
                                                       ▼
                                                    Blocked
```

| ステータス | 説明 | トリガー |
|-----------|------|---------|
| 📥 **Inbox** | 新規Issue | Issue作成時 |
| 📋 **Backlog** | 優先度付けされたタスク | トリアージ後 |
| ✅ **Ready** | 着手可能 | 依存関係解消後 |
| 🔨 **Development** | 開発中 | ブランチ作成時 |
| 🧪 **Verify** | CI/テスト検証中 | PR作成時 |
| 🚪 **Deploy Gate** | デプロイ承認待ち | STABLE達成時 |
| ✅ **Done** | 完了 | マージ後 |
| 🚫 **Blocked** | ブロック中 | 依存関係問題時 |

### 自動更新ルール

| イベント | アクション |
|---------|-----------|
| Issue作成 | → Inbox |
| ラベル付与 (priority) | → Backlog |
| ブランチ作成 | → Development |
| PR作成 | → Verify |
| CI全成功 + STABLE | → Deploy Gate |
| PR マージ | → Done |
| CI失敗 | → Development（差し戻し） |

## 🔄 自動開発フロー

### 全体フロー

```
ClaudeOS 起動（8時間タイマー開始）
    │
    ▼
[Monitor - 1h]
├── GitHub Projects 確認
├── 優先タスク選定
├── Issue → Ready ステータス更新
└── 開発計画策定
    │
    ▼
[Build - 2h]
├── WorkTree ブランチ作成
├── 設計・実装
├── ユニットテスト作成
└── Issue → Development 更新
    │
    ▼
[Verify - 2h]
├── pytest 実行
├── CI 確認（black/isort/flake8/bandit）
├── カバレッジ確認
├── STABLE 判定
└── Issue → Verify 更新
    │
    ├── 失敗 → CI Manager → Auto Repair → 再Verify
    │
    ▼ 成功
[Improve - 3h]
├── PR 作成
├── コードレビュー（自己レビュー）
├── ドキュメント更新
├── Issue → Deploy Gate 更新
├── マージ
└── Issue → Done 更新
    │
    ▼
次のタスクへ（時間が残っていれば）
```

### ドキュメント連動

全ドキュメントカテゴリの指針に従って自動開発を実行:

| ドキュメント | 用途 |
|-------------|------|
| 01_プロジェクト概要 | プロジェクト方針の確認 |
| 02_要件定義 | 要件の確認 |
| 03_アーキテクチャ設計 | 設計方針の遵守 |
| 04_API仕様 | API設計の準拠 |
| 05_開発ガイド | コーディング規約の遵守 |
| 06_テスト戦略 | テスト方針の遵守 |
| 07_セキュリティ | セキュリティ要件の遵守 |
| 08_運用・デプロイ | デプロイ手順の遵守 |
| 09_コンプライアンス | コンプライアンス要件の確認 |
| 10_計画・ロードマップ | 計画との整合性確認 |

## 🎯 優先順位

### Phase優先順位（来週以降）

| 順位 | Phase | 理由 |
|:----:|:-----:|------|
| 1️⃣ | **Phase 16** | 本番デプロイの前提条件（認証基盤） |
| 2️⃣ | **Phase 17** | コンプライアンス要件（ログ長期保存） |
| 3️⃣ | **Phase 18** | ユーザー価値向上（可視化） |
| 4️⃣ | **Phase 19** | 品質目標達成（カバレッジ90%） |
| 5️⃣ | **Phase 20** | 本番要件（パフォーマンス検証） |

### 同一Phase内の優先順位

```
各Phase内:
1. セキュリティに関わる変更 (最優先)
2. コア機能の実装
3. テスト作成
4. ドキュメント更新
5. リファクタリング
```

### WorkTree並列開発の活用

```
同時に進行可能なタスク:
├── main worktree: 安定版（マージ先）
├── worktree-A: Phase N の機能実装
├── worktree-B: Phase N のテスト作成
└── worktree-C: ドキュメント更新

ルール:
├── main 直接 push 禁止
├── PR 必須
├── CI 確認必須
└── STABLE 判定必須
```
