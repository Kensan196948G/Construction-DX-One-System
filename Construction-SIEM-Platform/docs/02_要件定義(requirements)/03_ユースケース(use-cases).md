## 🦠 UC-001: ランサムウェア検知・自動対応

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-001 |
| **ユースケース名** | ランサムウェア検知・自動対応 |
| **アクター** | 🔍 SOC アナリスト（主）、💻 IT 管理者（副） |
| **トリガー** | 短時間での大量ファイル暗号化パターン検知、既知ランサムウェア IoC マッチ |
| **前提条件** | ログ収集エージェントが稼働中、ルールエンジンにランサムウェアルールが定義済み |

### 📝 フロー

```
1. [トリガー] IoT エッジ/ファイルサーバーからファイル操作ログが Kafka に送信される
2. [正規化] Normalizer が CLF 形式に変換
3. [検知] ルールエンジンがランサムウェアパターン（大量rename/encrypt）を検知
4. [IoC照合] ThreatIntelManager が既知ランサムウェア IoC と照合
5. [アラート] セベリティ CRITICAL のアラートを生成
6. [通知] マルチチャネル（メール・Slack）で SOC アナリストに即座に通知
7. [プレイブック] ランサムウェア対応プレイブックを自動実行
   7a. 感染端末のネットワーク隔離指示
   7b. 関連プロセスの停止指示
   7c. バックアップ復元手順の提示
8. [インシデント] P1 インシデントを自動起票
9. [SOC対応] SOC アナリストがインシデントを確認・対応
10. [完了] 対応完了後、インシデントをクローズ
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 検知時間（MTTD） | ⏱ 3分以内 |
| アラート生成 | セベリティ CRITICAL |
| 通知 | 即時（メール + Slack） |
| プレイブック | 自動実行開始 |
| インシデント | P1 で自動起票 |

## 🔐 UC-002: ブルートフォース攻撃検知

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-002 |
| **ユースケース名** | ブルートフォース攻撃検知 |
| **アクター** | 🔍 SOC アナリスト |
| **トリガー** | 同一ソースからの短時間での大量認証失敗（閾値: 10回/5分） |
| **前提条件** | 認証ログが収集されている、ブルートフォースルールが定義済み |

### 📝 フロー

```
1. [トリガー] 認証サーバーから認証失敗ログが連続して送信される
2. [正規化] CLF 形式に変換、ソース IP で集約
3. [検知] ルールエンジンが閾値超過（10回/5分）を検知
4. [相関分析] EventCorrelator が同一 IP からの他の不審活動を相関
5. [アラート] セベリティ HIGH のアラートを生成
6. [通知] SOC アナリストに通知
7. [対応] アナリストがアラートを確認し、必要に応じて IP ブロックを実施
8. [記録] 対応内容を監査ログに記録
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 検知時間（MTTD） | ⏱ 5分以内 |
| アラート生成 | セベリティ HIGH |
| 相関情報 | 同一ソースの関連イベント付与 |
| 対応 | IP ブロック実施可能 |

## 📐 UC-003: CAD/BIM データ大量エクスポート検知

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-003 |
| **ユースケース名** | CAD/BIM データ大量エクスポート検知 |
| **アクター** | 🔍 SOC アナリスト |
| **トリガー** | 通常パターンを超える大量の CAD/BIM ファイルダウンロード・エクスポート |
| **前提条件** | ファイルアクセスログが収集されている、UEBA ベースラインが学習済み |

### 📝 フロー

```
1. [トリガー] ファイルサーバーから大量のファイルアクセスログが送信される
2. [正規化] CLF 形式に変換
3. [UEBA] ユーザー行動分析が通常パターンからの逸脱を検知
   - 通常: 1日10ファイル → 異常: 1時間で100ファイル
4. [評価] ファイル種別（.dwg, .rvt, .ifc 等）を確認し、機密レベルを評価
5. [アラート] セベリティ HIGH のアラートを生成
6. [通知] SOC アナリスト + 現場管理者に通知
7. [調査] アナリストがユーザーの正当性を確認
   7a. 正当 → アラートを resolved に更新
   7b. 不正 → インシデント起票、アカウント停止
8. [記録] 調査結果を記録
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 検知時間（MTTD） | ⏱ 15分以内 |
| アラート生成 | セベリティ HIGH |
| UEBA 情報 | 通常パターンとの乖離度付与 |
| 対応 | アカウント停止可能 |

## 📡 UC-004: IoT デバイス異常検知

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-004 |
| **ユースケース名** | IoT デバイス異常検知 |
| **アクター** | 🔍 SOC アナリスト、🏗 現場管理者 |
| **トリガー** | IoT デバイスの異常通信パターン、未知デバイスの接続、ファームウェア改ざん |
| **前提条件** | IoT エッジデバイスからのログ収集が稼働中 |

### 📝 フロー

```
1. [トリガー] IoT エッジデバイスから異常な通信パターンが検出される
   - 通常と異なる宛先への通信
   - 異常な通信量
   - 未登録デバイスの接続
2. [正規化] CLF 形式に変換
3. [ML検知] Isolation Forest モデルが異常スコアを算出
4. [アラート] セベリティに応じたアラートを生成
5. [通知] SOC アナリスト + 現場管理者に通知
6. [現場対応] 現場管理者が物理的なデバイス確認を実施
7. [SOC対応] アナリストがネットワーク側の調査を実施
8. [対応完了] デバイス隔離またはファームウェア更新
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 検知時間（MTTD） | ⏱ 10分以内 |
| アラート生成 | ML 異常スコア付与 |
| 通知先 | SOC + 現場管理者 |
| 対応 | デバイス隔離・更新 |

## 👤 UC-005: 内部不正行為（UEBA）検知

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-005 |
| **ユースケース名** | 内部不正行為検知 |
| **アクター** | 🔍 SOC アナリスト |
| **トリガー** | UEBA が通常行動パターンからの著しい逸脱を検知 |
| **前提条件** | 十分な期間の行動ベースラインが学習済み |

### 📝 フロー

```
1. [学習] UEBA モデルがユーザー毎の通常行動パターンを学習
   - アクセス時間帯
   - アクセスリソース
   - 操作量・頻度
2. [トリガー] 以下のいずれかを検知
   - 深夜の大量データアクセス
   - 通常アクセスしないリソースへの操作
   - 権限昇格の試行
3. [相関分析] EventCorrelator が関連イベントを相関
4. [キルチェーン] 偵察→権限昇格→データ収集のパターンを識別
5. [アラート] セベリティ HIGH/CRITICAL のアラートを生成
6. [通知] SOC アナリスト + CISO に通知
7. [調査] 詳細な行動分析レポートを確認
8. [対応] アカウント停止、フォレンジック調査開始
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 検知時間（MTTD） | ⏱ 15分以内 |
| 分析情報 | キルチェーンステージ、行動スコア |
| 通知先 | SOC + CISO |
| 対応 | アカウント停止・フォレンジック |

## 📶 UC-006: オフライン現場からのバッファ復帰

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-006 |
| **ユースケース名** | オフライン現場バッファ復帰 |
| **アクター** | 💻 IT 管理者 |
| **トリガー** | オフライン現場のネットワーク接続復帰 |
| **前提条件** | エッジデバイスにローカルバッファが設定済み |

### 📝 フロー

```
1. [断絶] ネットワーク障害により現場がオフラインになる
2. [バッファリング] エッジデバイスがイベントをローカルバッファに蓄積
3. [復帰] ネットワーク接続が復帰する
4. [自動送信] バッファされたイベントが時系列順に Kafka へ送信される
5. [正規化] 通常フローと同様に CLF 正規化が実行される
6. [遡及検知] 蓄積されたイベントに対してルールエンジン・ML が遡及的に検知を実行
7. [アラート] 遡及検知で脅威が見つかった場合、アラートを生成（タイムスタンプは元の発生時刻）
8. [確認] IT 管理者がバッファ復帰完了を確認
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| データ損失 | ゼロ（バッファ容量内） |
| 送信順序 | 時系列順を保持 |
| 遡及検知 | 全イベントに対して実行 |
| タイムスタンプ | 元の発生時刻を保持 |

## 📄 UC-007: インシデント事後報告書生成

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-007 |
| **ユースケース名** | インシデント事後報告書生成 |
| **アクター** | 🔍 SOC アナリスト、🛡 CISO |
| **トリガー** | インシデントのクローズ、または CISO からのレポート要求 |
| **前提条件** | インシデントが記録済み、関連アラート・イベントが紐付け済み |

### 📝 フロー

```
1. [トリガー] SOC アナリストがインシデントをクローズ、またはレポート生成を要求
2. [データ収集] インシデントに関連する以下の情報を自動収集
   - インシデント概要（起票〜クローズまでのタイムライン）
   - 関連アラート一覧
   - 実行されたプレイブック
   - 対応履歴
3. [レポート生成] POST /api/v1/reports/generate で報告書を自動生成
4. [レビュー] SOC アナリストが内容を確認・補足
5. [承認] CISO がレポートを承認
6. [配布] 関係者へレポートを配布
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| 生成時間 | 30秒以内 |
| 内容 | タイムライン、原因分析、対応履歴、再発防止策 |
| 形式 | 構造化 JSON / レポートビュー |
| 監査対応 | ISO 20000 準拠 |

## ✅ UC-008: コンプライアンスチェック実行

### 概要

| 項目 | 内容 |
|------|------|
| **ユースケースID** | UC-008 |
| **ユースケース名** | コンプライアンスチェック実行 |
| **アクター** | 🛡 CISO、💻 IT 管理者 |
| **トリガー** | 定期チェック（月次）、または手動実行要求 |
| **前提条件** | チェック項目が定義済み |

### 📝 フロー

```
1. [トリガー] CISO が GET /api/v1/compliance/check を実行
2. [チェック実行] 以下の項目を自動チェック
   - ISO 27001 A.8.15: ログ収集の網羅性
   - ISO 27001 A.8.16: 監視活動の実施状況
   - NIST CSF DE.CM: 継続的モニタリングの稼働状況
   - アクセス制御の設定確認
   - 監査ログの保存期間確認
3. [スコアリング] 各項目の準拠状況をスコアリング
4. [レポート生成] 準拠状況レポートを生成
5. [是正] 非準拠項目に対する是正勧告を提示
6. [追跡] 是正状況を追跡
```

### ✅ 期待結果

| 項目 | 期待値 |
|------|--------|
| チェック時間 | 1分以内 |
| カバレッジ | 全準拠規格をチェック |
| 出力 | スコア、準拠/非準拠一覧、是正勧告 |
| 追跡 | 是正状況の追跡可能 |
