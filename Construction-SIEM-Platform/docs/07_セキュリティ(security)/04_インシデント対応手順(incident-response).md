## 🔍 Phase 1: 検知

### 検知ソース

| 検知エンジン | 検知内容 | 優先度 |
|-------------|---------|:------:|
| 🎯 Sigmaルールエンジン | 既知の攻撃パターン | 高 |
| 🤖 ML異常検知 (Isolation Forest) | 未知の異常行動 | 中 |
| 🕵️ IoC照合エンジン | 既知の脅威指標マッチ | 高 |
| 📊 閾値ベースアラート | メトリクス異常 | 中 |
| 👤 ユーザー報告 | 手動報告 | 可変 |

### 検知からアラート生成まで

```
イベント受信（Kafka）
    │
    ├──► Sigmaルール照合 ──► ルールマッチ → アラート生成
    │
    ├──► ML異常検知 ──► 異常スコア > 閾値 → アラート生成
    │
    └──► IoC照合 ──► IoCマッチ → アラート生成

アラート生成
    │
    ├── severity 自動判定
    ├── 関連イベント収集
    └── SOC通知（Grafana Alert / Webhook）
```

### アラート構造

```json
{
  "alert_id": "ALT-20260324-001",
  "detection_source": "sigma_rule",
  "rule_id": "construction_unauthorized_access",
  "severity": 8,
  "title": "建設現場IoTデバイスへの不正アクセス検知",
  "description": "現場カメラシステムへの異常なログイン試行を検出",
  "source_ip": "203.0.113.50",
  "target": "site-camera-01.construction.local",
  "timestamp": "2026-03-24T10:30:00Z",
  "related_events": ["EVT-001", "EVT-002", "EVT-003"]
}
```

## 🏷 Phase 2: トリアージ

### severity判定基準

| severity | 範囲 | 分類 | 説明 |
|:--------:|:----:|:----:|------|
| 9-10 | 🔴 | **P1 - Critical** | 安全リスク、データ漏洩確定、システム全停止 |
| 7-8 | 🟠 | **P2 - High** | 重要システム侵害、大規模異常 |
| 4-6 | 🟡 | **P3 - Medium** | 単一システム異常、疑わしい活動 |
| 1-3 | 🟢 | **P4 - Low** | 情報提供レベル、軽微な異常 |

### 優先度別対応

| 優先度 | 初動時間 | 対応完了目標 | 担当者 | 通知先 |
|:------:|:--------:|:----------:|--------|--------|
| 🔴 P1 | **15分以内** | **4時間以内** | SOCリーダー + 全アナリスト | CISO、現場責任者、経営層 |
| 🟠 P2 | **30分以内** | **8時間以内** | 担当アナリスト + リーダー | SOCリーダー、IT管理者 |
| 🟡 P3 | **2時間以内** | **24時間以内** | 担当アナリスト | SOCリーダー |
| 🟢 P4 | **8時間以内** | **72時間以内** | 担当アナリスト | チケットのみ |

### トリアージチェックリスト

```
□ アラートの信頼性確認（誤検知チェック）
□ 影響範囲の初期評価
  □ 影響を受けるシステム数
  □ 影響を受けるユーザー数
  □ 建設現場の安全への影響有無
□ severity / 優先度の決定
□ 担当者のアサイン
□ チケット作成（GitHub Issues）
□ 必要に応じてエスカレーション
```

## ⚙️ Phase 3: 対応

### プレイブック自動実行

| プレイブック | トリガー | 自動アクション |
|-------------|---------|--------------|
| `PB-001` 不正アクセス対応 | 認証失敗 > 10回/分 | IP一時ブロック、アカウントロック |
| `PB-002` マルウェア検知対応 | IoC (hash) マッチ | 端末隔離通知、スキャン指示 |
| `PB-003` IoTデバイス異常 | IoT異常パターン検知 | デバイス監視強化、現場通知 |
| `PB-004` データ漏洩対応 | 大量データ転送検知 | ネットワーク制限、調査開始 |
| `PB-005` DoS攻撃対応 | レート制限連続超過 | WAFルール強化、ISP連絡 |

### プレイブック実行フロー

```
アラート生成
    │
    ▼
プレイブック選択
    │
    ├── 自動実行アクション
    │   ├── レート制限強化
    │   ├── アカウントロック
    │   └── 通知送信
    │
    ├── 手動承認必要アクション（P1時）
    │   ├── ネットワーク隔離
    │   ├── サービス停止
    │   └── 外部機関への報告
    │
    └── SOC通知
        ├── Grafana Alert
        ├── Webhook（Teams/Slack）
        └── メール（P1/P2）
```

### SOC通知内容

```

アラートID: ALT-20260324-001

検知エンジン: Sigma Rule (construction_unauthorized_access)

攻撃元IP: 203.0.113.50
プレイブック: PB-001 自動実行中

対応URL: https://siem.example.com/incidents/ALT-20260324-001
```

## 🔒 Phase 4: 封じ込め

### 封じ込め手順

| アクション | 条件 | 承認 | 実行方法 |
|-----------|------|:----:|---------|
| 🔐 アカウント無効化 | 侵害アカウント特定時 | 自動（P1/P2） | API経由でアカウントロック |
| 🌐 IPブロック | 攻撃元IP特定時 | 自動 | ファイアウォールルール追加 |
| 🔌 ネットワーク隔離 | 侵害端末特定時 | **手動承認** | VLAN変更 / ポート無効化 |
| 🏗 現場システム隔離 | IoT侵害時 | **手動承認** | エッジルーター設定変更 |
| 🛑 サービス停止 | 重大侵害時 | **CISO承認** | docker compose stop |

### 封じ込め判断フロー

```
侵害確認
    │
    ├── 攻撃進行中？
    │   ├── Yes → 即座に封じ込め（自動/手動）
    │   └── No  → 証拠保全後に封じ込め
    │
    ├── 影響範囲
    │   ├── 単一端末 → 端末隔離
    │   ├── セグメント → VLAN隔離
    │   └── 全体 → サービス停止検討
    │
    └── 建設現場安全影響
        ├── あり → 現場責任者に即連絡 + 物理的対策
        └── なし → IT対応のみ
```

## 🔧 Phase 5: 復旧

### 影響範囲分析

```
□ 侵害タイムラインの確定
  □ 初期侵入時刻
  □ 横展開の有無
  □ データアクセス範囲
□ 影響を受けたシステムの一覧作成
□ 影響を受けたデータの特定
□ 影響を受けたユーザーの特定
```

### 是正措置

| 措置 | 内容 | 担当 |
|------|------|------|
| パスワードリセット | 侵害アカウントの全パスワード変更 | IT管理者 |
| パッチ適用 | 脆弱性が原因の場合、即座にパッチ適用 | インフラチーム |
| 設定修正 | 設定ミスが原因の場合、設定変更 | インフラチーム |
| IoC追加 | 新たに判明したIoCをフィードに追加 | アナリスト |
| ルール追加 | 再発防止のSigmaルール追加 | アナリスト |
| 監視強化 | 一定期間の監視レベル引き上げ | SOC |

### 復旧確認チェックリスト

```
□ 封じ込めた脅威が完全に排除されたことを確認
□ システムが正常に動作していることを確認
  □ /health/detailed で全コンポーネント正常
  □ ログ収集が再開されていることを確認
□ 監視が正常に機能していることを確認
□ ユーザーへの影響が解消されたことを確認
□ 復旧完了を関係者に通知
```

## 📞 エスカレーション

### エスカレーションマトリクス

| 条件 | エスカレーション先 | 方法 | タイミング |
|------|------------------|------|-----------|
| P1インシデント発生 | CISO + 経営層 | 電話 + メール | 即座 |
| P2が4時間未解決 | SOCマネージャー | メール + チャット | 4時間後 |
| P3が24時間未解決 | SOCリーダー | チャット | 24時間後 |

## ⏱ SLA定義

### 対応SLA

| 優先度 | 初動開始 | 封じ込め | 復旧完了 | 事後報告 |
|:------:|:--------:|:-------:|:-------:|:-------:|
| 🔴 P1 | 15分 | 1時間 | 4時間 | 24時間 |
| 🟠 P2 | 30分 | 4時間 | 8時間 | 3日 |
| 🟡 P3 | 2時間 | 8時間 | 24時間 | 1週間 |
| 🟢 P4 | 8時間 | 24時間 | 72時間 | 2週間 |

### SLA測定指標

| 指標 | 定義 | 目標 |
|------|------|------|
| **MTTD**（平均検知時間） | イベント発生〜アラート生成 | < 15分 |
| **MTTR**（平均対応時間） | アラート生成〜復旧完了 | < 2時間 |
| **MTTC**（平均封じ込め時間） | アラート生成〜封じ込め完了 | < 1時間（P1） |
| **誤検知率** | 誤検知アラート / 全アラート | < 5% |

### SLAモニタリング

```
SLA超過時の自動アクション:
├── 初動SLA超過 → 自動エスカレーション
├── 封じ込めSLA超過 → マネージャー通知
├── 復旧SLA超過 → CISO通知
└── 週次SLAレポート自動生成
```
